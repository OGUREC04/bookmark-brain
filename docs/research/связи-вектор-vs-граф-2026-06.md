# Связи между заметками: вектор vs граф vs гибрид — ресёрч

*Дата: 2026-06-01 · Источников: 12 · Уверенность: высокая (есть прямой продуктовый аналог)*
*Для: Phase 5A «Connections layer» bookmark-brain · Входит в будущий PRD docs/prd/CONNECTIONS-MVP.md*

## Executive summary

Тезис «графы работают лучше» — **правда, но не для нашей задачи.** Графовые БД (Neo4j) и GraphRAG выигрывают на **многошаговых рассуждениях по связям между сущностями** (3+ объекта, цепочки зависимостей, синтез по большому корпусу). Наша задача — **автоматически связать похожие по смыслу заметки, показать «связанное» и дать семантический поиск** — это классическая работа для векторов, а не графов.

**Рекомендация для MVP:** дешёвый гибрид — **вектор рождает связи (косинус), рёбра храним в PostgreSQL (`bookmark_links`), графовые фичи (двусторонность, обход в 2 шага, визуализация) делаем обычным SQL.** Отдельный графовый движок (Neo4j) и GraphRAG для MVP **не нужны** — это инфраструктура и расходы под задачи, которых у нас нет. У нас уже есть pgvector + эмбеддинги + HNSW + прототип поиска похожих в дедупликаторе → MVP почти не требует новой инфраструктуры.

Прямой продуктовый аналог — **rhizome** (semantic backlinks для Obsidian/Logseq) и **Smart Connections** — делают ровно это: локальные эмбеддинги + косинус + порог ~0.75 + top-5 + чанкинг длинных заметок. Без графовой БД.

---

## 1. Вектор vs граф: решают разные задачи

Вектор отвечает на вопрос «**что похоже на это?**», граф — «**что связано с этим?**». Это фундаментальное различие, которое определяет выбор ([Zilliz: pgvector vs Neo4j](https://zilliz.com/blog/pgvector-vs-neo4j-a-comprehensive-vector-database-comparison), [bix-tech](https://bix-tech.com/vector-databases-compared-pinecone-vs-pgvector-vs-neo4j-and-when-to-use-each/)).

- **Вектор (pgvector kNN):** каждая заметка → эмбеддинг (у нас Voyage voyage-3, 1024). Связь = близость по косинусу. Связи **рождаются сами** из смысла, без ручной разметки. Минус: знает только «похоже/не похоже», не знает тип связи и «почему».
- **Граф (Neo4j):** заметки — узлы, связи — типизированные рёбра. Силён в **обходе** (что в 2 шагах, какой узел — мост, кратчайший путь) и многошаговом рассуждении. Минус: **сам связи не создаёт** — рёбра кто-то должен наполнить (в Obsidian — пользователь руками через `[[ссылку]]`; у нас руками никто не ставит).

Ключ, который обычно недоговаривают: связи нужно сначала **родить** (это умеет только вектор), и лишь потом по ним можно **ходить** (это умеет граф). Поэтому правильный ответ — не «или-или», а гибрид ([Glean: knowledge graph vs vector](https://www.glean.com/blog/knowledge-graph-vs-vector-database)).

## 2. Нужен ли Neo4j сверх рёбер в PostgreSQL

Для наших задач (двусторонние связи, обход в 2 шага, визуализация графа) — **нет.**

- **Когда pgvector + Postgres достаточно:** «вы уже на Postgres, масштаб умеренный, нужен векторный поиск внутри SQL без новой инфраструктуры» ([bix-tech](https://bix-tech.com/vector-databases-explained-pinecone-pgvector-and-neo4j-plus-how-to-choose/)). Это в точности наш случай.
- **Когда реально нужен граф-движок:** агентам нужно рассуждать по цепочкам отношений — зависимости сущностей, временные последовательности, иерархии, причинно-следственные цепи; на чистом векторе они упираются в стену, потому что эмбеддинги кодируют **смысл, а не структуру** ([AgentMarketCap, апр 2026](https://agentmarketcap.ai/blog/2026/04/07/graph-rag-vs-vector-rag-agent-memory-neo4j-pgvector)).
- **Честная оговорка:** глубокий обход графа тяжело «прикрутить» к реляционной БД — Cypher, графовый движок и community-detection алгоритмы Neo4j не воспроизводятся расширением Postgres ([Zilliz](https://zilliz.com/blog/pgvector-vs-neo4j-a-comprehensive-vector-database-comparison)). Но обход в **2 шага** на десятках тысяч заметок — это тривиальный рекурсивный SQL по `bookmark_links`, а не «глубокий обход».

**Практический совет из источников:** «Начни с pgvector на существующем Postgres, измерь точность. Если на сложных запросах плохо — пойми, это проблема *поиска* или *рассуждения*. Если рассуждения — тогда добавляй граф» ([bix-tech](https://bix-tech.com/vector-databases-compared-pinecone-vs-pgvector-vs-neo4j-and-when-to-use-each/)).

**Визуализация графа** (твой таб «как в Obsidian») графовую БД **не требует**: это фронтенд — force-directed рендер рёбер из `bookmark_links` (d3-force / react-force-graph). Рёбра живут в Postgres, отдаются по API, рисуются на клиенте.

## 3. Как «связать всё», не взрывая базу на N² рёбер

Связывать каждую-с-каждой нельзя (N² рёбер + шум). Решение — **kNN-граф с потолком k**: у каждой заметки храним только top-k ближайших.

- **k:** на практике берут небольшое k (единицы–десятки). Для отображения хватает 5; для хранения разумно держать запас (k≈20–50), чтобы «Посмотреть все связи» имело глубину ([Princeton: kNN graph construction](https://www.cs.princeton.edu/cass/papers/www11.pdf), [Springer: kNN review 2024](https://link.springer.com/article/10.1186/s40537-024-00973-y)).
- **Порог similarity:** ребро создаём только если косинус выше порога. Практика по эмбеддингам: 0.85 — высокая уверенность, 0.80 — кандидат, ниже 0.80 — слабо ([Anil's notes](https://anilktalla.medium.com/cosine-similarity-my-notes-5bc399a2d9fc)). rhizome для родственной задачи (related notes) использует **0.75 = medium**, 0.60 = low, 0.88 = high ([rhizome](https://github.com/matzalazar/rhizome)).
- **mutual-kNN** (ребро только если A в top-k у B И B в top-k у A) заметно режет шум — рекомендуется, когда нужны «самые значимые» связи ([arxiv 0912.3408](https://arxiv.org/pdf/0912.3408)).
- **Порог — калибруется итеративно** на реальных данных: построить распределение косинусов на выборке и найти «sweet spot» ([Anil's notes](https://anilktalla.medium.com/cosine-similarity-my-notes-5bc399a2d9fc)).

**Для нас:** храним top-k (≈20–50) с порогом ~0.75 (стартовое значение, калибруем на бэкфилле), опционально mutual-kNN для чистоты; показываем top-5 + «Посмотреть все».

## 4. Как делают реальные продукты

Продукты, делающие **ровно нашу фичу** (авто-связи похожих заметок), используют вектор, не граф:

- **rhizome** (Obsidian/Logseq): локальный multilingual sentence-transformer → косинус → порог 0.75 → top-5 → секция `## Related Notes` как wikilinks. Длинные заметки (>512 токенов) бьются на перекрывающиеся чанки, эмбеддинги усредняются. Малый корпус — точный numpy-поиск, большой — HNSW ([rhizome](https://github.com/matzalazar/rhizome)). **Это буквально наш дизайн.**
- **Smart Connections** (Obsidian): локальные эмбеддинги, ранжирует заметки/блоки против текущей заметки, «что ещё в хранилище связано с тем, что я пишу сейчас», работает офлайн ([Smart Connections](https://smartconnections.app/smart-connections/)).
- **GraphRAG (Microsoft)** и графовые подходы — это другой класс задач (QA-синтез и multi-hop по большому корпусу), не «покажи похожие заметки».

Вывод: для «связать похожее + семантический поиск + показать связанное + полистать граф» индустрия использует **вектор + простые рёбра**, а граф-движок подключают под multi-hop рассуждения.

## 5. Проактивные связи на сохранении: токены

- **Создание связей при сохранении = 0 токенов LLM.** Это чистая математика: посчитать косинус нового эмбеддинга против существующих (HNSW-поиск в Postgres). Эмбеддинг заметки и так считается в pipeline. Никаких вызовов LLM.
- **Где LLM добавляет ценность:** только **типизация связи** — пометить «продолжает / противоречит / пример к / та же тема». Это требует вызова LLM на пару заметок (~сотни токенов на пару) и для MVP **избыточно**. Откладываем.
- GraphRAG-стиль (LLM извлекает сущности и строит граф) для personal-notes — дорого и не нужно: полное GraphRAG-индексирование стоит $$ ([buildmvpfast](https://www.buildmvpfast.com/blog/graphrag-vs-vector-rag-knowledge-graph-ai-2026)); ценность — на multi-hop QA, не на нашей задаче.

## 6. Что в MVP, что отложить

**В MVP (на текущем стеке, почти без новой инфраструктуры):**
- Таблица `bookmark_links` (рёбра kind='similar', weight=cosine) — уже в эскизе SMART-SPACES-MVP.
- На save: top-k similar по косинусу (HNSW), порог ~0.75, запись рёбер. 0 токенов.
- Разовый бэкфилл существующих заметок.
- Двусторонность: запрос связей в обе стороны (`from_id = X OR to_id = X`).
- API `GET /bookmarks/{id}/related?limit=5` + «Посмотреть все».
- Чанкинг длинных заметок (>512 токенов) с усреднением эмбеддингов — иначе длинные тексты искажают близость.
- Семантический поиск по смыслу — отдельный косинус-запрос (у нас уже есть гибридный score = α·cosine + β·fts).

**Отложить (следующие фазы):**
- Визуализация-граф (таб «как в Obsidian») — фронтенд поверх рёбер, без новой БД.
- Обход в 2 шага (навигация по графу связей) — рекурсивный SQL.
- Типизация связей через LLM (продолжает/противоречит) — когда понадобится «почему связано».
- Graph-движок / GraphRAG — **только если** появятся multi-hop запросы и измерения покажут провал векторного поиска. Не раньше.

## Key takeaways

1. **«Графы лучше» верно для multi-hop рассуждений, не для авто-связывания похожего.** Наша задача — векторная.
2. **Гибрид нужен, но дешёвый:** вектор рождает рёбра → рёбра в Postgres → графовые фичи через SQL. Neo4j/GraphRAG для MVP не нужны.
3. **Проактивные связи = 0 токенов** (чистый косинус). LLM только для типизации связи — отложить.
4. **Параметры из практики:** порог ~0.75 (калибровать), хранить top-k≈20–50, показывать 5, mutual-kNN против шума, чанкинг длинных заметок.
5. **Прямой аналог (rhizome) подтверждает наш дизайн** почти буквально.
6. **Риск «понять, что нужен граф, на 6 месяцев позже» (для нас низкий):** возникает на multi-hop QA по большому корпусу; для personal note-linking порог не достигается. Заложим метрику recall, чтобы поймать, если ошиблись.
