# БТ-7: Поиск и просмотр

> Бизнес-требования · living-doc «как ведёт себя сейчас»
> Обновлено: 2026-06-01

## Назначение и границы

Покрывает: поиск по сохранённому (как из бота, так и из Mini App), AI-саммари
по результатам поиска («ответ по сохранённому»), просмотр одной карточки,
случайную закладку (`/random`).

НЕ покрывает: захват и сохранение — [БТ-1](бт-01-захват.md); классификацию,
теги, summary самой закладки — [БТ-2](бт-02-ai-обработка.md); голосовую
маршрутизацию интентов (как голос попадает в поиск vs напоминание) подробно —
[БТ-5](бт-05-напоминания.md). Уборка/удаление из карточки — [БТ-9](бт-09-уборка.md).

## Пользовательские истории

Как пользователь:
1. Я хочу написать своими словами «что я читал про rlhf» и найти закладку, даже если в ней нет слова «rlhf».
2. Я хочу искать по точному слову («идли») и находить, даже когда AI ещё не разметил закладку.
3. Я хочу короткий ответ-выжимку по найденному, а не только список ссылок.
4. Я хочу открыть карточку и увидеть заголовок, саммари, теги, полный текст и ссылку.
5. Я хочу нажать `/random` и достать случайную старую закладку, чтобы вспомнить, что у меня вообще лежит.
6. Я хочу искать голосом — сказать «найди про дизайн» и получить результаты.

## Проблема / зачем так

Память без поиска — свалка. Но точное совпадение слов ловит мало: человек
сохранил тред про «обучение моделей с подкреплением», а ищет «rlhf» — текстовый
поиск промахнётся. Поэтому поиск гибридный: семантика (близость по смыслу через
эмбеддинги) и полнотекстовый (точные слова) считаются вместе и складываются с
весами. Короткий запрос («идли») — это чаще точное слово, и текстовый вес растёт;
длинный запрос-фраза — это смысл, и вес уходит в семантику.

Отдельная боль: список из десяти ссылок всё равно надо открывать по одной.
Поэтому по верхушке результатов LLM собирает выжимку в 2-4 предложения с
маркерами `[1]`, `[2]` — как one-box у Google. Если LLM упал или результаты
слабые — саммари просто нет, выдача показывается без него.

## Поведение (правила)

### Как считается релевантность

Гибрид из двух сигналов, веса зависят от длины запроса:

| Запрос | Семантика | Текст |
|--------|-----------|-------|
| ≤ 2 слов | 0.5 | 0.5 |
| > 2 слов | 0.7 | 0.3 |

- **Семантика** — косинусная близость эмбеддинга запроса к эмбеддингу закладки (pgvector `<=>`). Если у закладки эмбеддинга нет — её семантический вклад 0.
- **Текст** — `ts_rank` по полнотекстовому индексу (`plainto_tsquery('russian', …)`).
- Итоговый score = `вес_сем × семантика + вес_текст × текст`. Сортировка по убыванию.
- В выдачу попадают только записи, где **хотя бы один** сигнал > 0.

### Деградация и fallback

| Ситуация | Поведение |
|----------|-----------|
| Эмбеддинг запроса не получился (API упал) | переключается на чистый full-text (веса 0/1), молча |
| Ни семантика, ни текст ничего не дали | финальный fallback — `ILIKE` по `raw_text` / `title` / `url`, score 0.1, сортировка по дате |
| Совсем ничего не нашлось | пустая выдача, total = 0 |

`ILIKE`-fallback ловит сырой текст ещё не размеченных AI-закладок и точные
подстроки, которые проиграли по рангу.

### Лимиты выдачи

| Точка входа | Запрашивает | Показывает |
|-------------|-------------|------------|
| Бот `/search` | top-5 | до 5, при `total > 5` — «…и ещё N. Уточни запрос» |
| Голосовой поиск | top-5 | то же |
| Mini App | top-20 | все полученные |

Backend по умолчанию `limit=20`, максимум 50, есть `offset` (пагинация
API-уровня; бот её не использует — всегда первая страница).

### AI-саммари по результатам («ответ по сохранённому»)

Считается по верхушке выдачи. Не делается вовсе, если:

| Условие отказа от саммари | Порог |
|---------------------------|-------|
| Запрос слишком короткий | < 2 слов |
| Результатов нет | — |
| Лучший результат слабый | score < 0.15 |
| LLM упал / вернул пусто | — |

Когда делается: берёт top-5, скармливает их заголовки + суть/идеи модели,
получает 2-4 предложения с маркерами `[1]..[5]`. Если фактов в выжимках нет —
модели велено честно сказать «В сохранённых закладках нет ответа на этот запрос».
Бот `/search` саммари **не запрашивает** (шлёт только query+limit) — он показывает
список без one-box. Mini App саммари получает (backend-дефолт `with_summary=true`)
и рисует блок «✦ ответ по сохранённому».

### Просмотр карточки

- **Бот** (`view:` callback из `/list`, `/lists`): заголовок, категория, теги,
  статус AI-обработки, саммари, ссылка, полный текст (обрезается до 3000 символов),
  кнопки «🗑 Удалить» / «◀️ К списку». Всё сообщение режется до 4000 символов (лимит Telegram).
- **Mini App** (`DetailScreen`): шапка «host · дата», при незавершённой обработке —
  «ai обрабатывает…»; заголовок, summary курсивом, список задач (если это
  task_list — редактируемый), полный текст, кнопка «открыть источник», теги,
  блок «✦ ключевые мысли».

### Случайная закладка (`/random`)

Только бот. Backend отдаёт одну закладку через `ORDER BY random() LIMIT 1`.
Бот показывает заголовок, категорию, summary (или первые 150 символов сырого
текста), теги, ссылку. Пустая база → 404 → «У тебя пока нет сохранённых закладок.»

## Use Cases

### UC-7.1. Поиск из бота `/search`

- **Триггер:** `/search статьи про дизайн`
- **Основной сценарий:**
  1. Бот шлёт «печатает…», запрашивает top-5.
  2. Гибридный поиск возвращает результаты.
  3. Бот: `Результаты по запросу "статьи про дизайн" (3 найдено):` + нумерованный список (заголовок, summary, теги, «Открыть», «Релевантность: 87%»).
- **Альтернативные сценарии:**
  - A1. `/search` без запроса → «Напиши запрос после /search. Пример: …».
  - A2. Найдено больше 5 → в конце «…и ещё N. Уточни запрос для точных результатов.»
  - A3. Ничего не найдено → `Ничего не найдено по запросу "…"`.
  - A4. API упал → «Ошибка поиска. Попробуй позже.»
- **Результат:** список до 5 результатов. Саммари бот не показывает.

### UC-7.2. Поиск в Mini App

- **Предусловия:** открыт экран «поиск».
- **Триггер:** пользователь печатает «анализ rlhf».
- **Основной сценарий:**
  1. Через 350 мс после остановки ввода (или сразу по Enter) уходит запрос top-20.
  2. Пока ждём — пульсирующий индикатор.
  3. Показывается блок «✦ ответ по сохранённому» (если саммари пришло) + карточки результатов (host, дата, заголовок, summary).
- **Альтернативные сценарии:**
  - A1. Поле пустое → стартовый EmptyState «о чём подумать?» с примерами, поиск не идёт.
  - A2. Ноль результатов → EmptyState «∅ ничего не нашлось / попробуй другие слова».
  - A3. Запрос < 2 слов → саммари не приходит, только список.
  - A4. Быстро меняешь запрос → ответ устаревшего запроса отбрасывается (по reqId), показывается только последний.
  - A5. Тап по карточке результата → тост «в разработке» (открытие карточки из поиска ещё не подключено; деталь открывается из ленты «мысли»).
- **Результат:** список + опциональный one-box.

### UC-7.3. Голосовой поиск

- **Предусловия:** голосовое распознано как поисковый интент (короткое, «найди …», «где у меня …», вопросительное).
- **Триггер:** голос «найди про even».
- **Основной сценарий:**
  1. Бот reply’ит `🔍 <распознанный текст>` (прозрачность STT).
  2. Запрашивает top-5, показывает `🔍 Результаты по запросу "<…>" (N найдено):` + список.
- **Альтернативные сценарии:**
  - A1. Очищенный запрос пуст → «Не удалось распознать поисковый запрос.»
  - A2. Ноль результатов → `Ничего не найдено по запросу "…"`.
  - A3. «напомни что я покупал» — поисковая форма «напомни что/про/какие» → уходит в поиск, не в напоминание (см. [БТ-5](бт-05-напоминания.md)).
- **Результат:** список до 5.

### UC-7.4. AI-саммари по результатам

- **Предусловия:** запрос ≥ 2 слов, есть результат с score ≥ 0.15, `with_summary` включён (Mini App).
- **Триггер:** поиск «как устроен rlhf».
- **Основной сценарий:**
  1. Поиск даёт результаты.
  2. LLM по top-5 пишет 2-4 предложения с маркерами `[1][2]`.
  3. Mini App рисует блок «✦ ответ по сохранённому».
- **Альтернативные сценарии:**
  - A1. Выжимки не отвечают на запрос → текст «В сохранённых закладках нет ответа на этот запрос».
  - A2. LLM упал / вернул пусто → саммари нет, выдача показывается как есть.
  - A3. Запрос 1 слово или лучший score < 0.15 → саммари не генерируется.

### UC-7.5. Просмотр карточки

- **Триггер:** нажатие на закладку (бот: `view:` из списка; Mini App: тап из ленты «мысли»).
- **Основной сценарий:**
  1. Загружается полная закладка.
  2. Бот: заголовок, категория, теги, статус AI, саммари, ссылка, полный текст + кнопки.
  3. Mini App: шапка, заголовок, summary, текст, источник, теги, «ключевые мысли».
- **Альтернативные сценарии:**
  - A1. Закладка ещё в обработке → бот показывает статус (`⏳`/`🕐`), Mini App — «ai обрабатывает…».
  - A2. Это task_list → Mini App рисует редактируемый список задач (см. [БТ-4](бт-04-списки-задач.md)).
  - A3. Очень длинный текст → бот режет до 3000 символов («… текст обрезан»), всё сообщение до 4000.
  - A4. Загрузка не удалась → бот: «Ошибка загрузки».

### UC-7.6. Случайная закладка `/random`

- **Триггер:** `/random`
- **Основной сценарий:**
  1. Backend отдаёт случайную закладку.
  2. Бот: заголовок, категория, summary, теги, ссылка.
- **Альтернативные сценарии:**
  - A1. База пуста (404) → «У тебя пока нет сохранённых закладок.»
  - A2. API упал → «Ошибка. Попробуй позже.»
- **Результат:** одна случайная закладка. Дубли между вызовами возможны (честный random, без памяти показанного).

### UC-7.7. Правка тела текста заметки (Mini App, тикет 0rn)

- **Предусловия:** открыт `DetailScreen` любой заметки (ссылка с комментарием, расшифровка голосового, обычный текст). Каноничное редактируемое поле — `raw_text` (для голосовых тоже `raw_text`, исходная `transcription` не трогается).
- **Триггер:** пользователь тапает по тексту → правит на месте → правка сохраняется автоматически (`PATCH /bookmarks/{id}` с `raw_text`) при уходе из поля (blur), без кнопок «Изменить/Сохранить» (модель Apple Notes).
- **Основной сценарий:**
  1. Backend сохраняет новый текст (триммит).
  2. Оценивает масштаб правки (похожесть старого/нового текста).
  3. Если правка **значимая** (мог поменяться смысл) → `ai_status=pending` + фоновая полная переобработка (embedding + summary/key_ideas/title/теги). Mini App показывает «Brain переосмысливает…» и обновляет карточку, когда `ai_status` станет терминальным.
  4. Если правка **мелкая** (пара слов) → текст сохранён, переобработки нет, `ai_status` остаётся терминальным; фронт сразу показывает финал.
- **Альтернативные сценарии:**
  - A1. Правка заметки к ссылке → меняется только тело текста, `url` не ломается.
  - A2. Правка расшифровки голосового → редактируется `raw_text` (поиск/embedding по нему); поле `transcription` (оригинал STT) сохраняется как есть.
  - A3. Пустой текст после trim → `422`, Mini App показывает ошибку, не отправляет.
  - A4. Чужая заметка → `404` (user-scoped).
  - A5. Уход с экрана во время правки (системная/Telegram «Назад», переход на связанную заметку) → черновик сохраняется при размонтировании экрана (flush), т.к. blur не срабатывает; дедуп не даёт отправить тот же текст дважды (без двойной AI-переобработки).
- **Правила:** порог значимости адаптивен к длине — в короткой заметке даже пара слов («не …») считается значимой и переобрабатывается, в длинной — нет. Это экономит холостую LLM-перегенерацию. После переобработки поиск находит по новому тексту (`search_vector` триггер БД + новый embedding). `task_list`-заметки тело так не редактируют — у них свой структурный редактор (см. [БТ-4](бт-04-списки-задач.md)).
