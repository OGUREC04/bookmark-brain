# БТ-3: Дедупликация

> Бизнес-требования · living-doc «как ведёт себя сейчас»
> Обновлено: 2026-05-24
> Связано: ADR (нет отдельного) · код `backend/app/services/dedup_checker.py`, `backend/app/worker/processing.py`, `backend/app/worker/dedup.py`, `bot/handlers/tasks/dedup.py`, `backend/app/services/reminder_creator.py`

## Назначение и границы

Покрывает: как система ловит повторно присланную закладку/список, что бот при этом пишет
и как пользователь выбирает судьбу дубля (открыть / удалить / обновить / оставить оба).
Плюс — какие сообщения дедупликация **намеренно пропускает** (напоминания, незавершённые
списки до подтверждения).

НЕ покрывает: точечную дедупликацию напоминаний «тот же текст + та же минута» (E15) —
её механика и тексты подтверждения описаны в [БТ-5](бт-05-напоминания.md) и
[БТ-6](бт-06-напоминания-списки.md). Здесь — только факт пропуска general-дедупа для
reminder-намерений. Слияние похожих списков задач («🔄 Похожий список … Объединить?») —
в [БТ-4](бт-04-списки-задач.md); здесь описана только точка, где дедуп его запускает.

## Пользовательские истории

Как пользователь:
1. Я хочу, чтобы повторно скинутая ссылка не превращалась во вторую идентичную закладку без предупреждения.
2. Я хочу сам решать, что делать с дублем: открыть старое, удалить новое, перезаписать старое или оставить обе записи.
3. Я хочу, чтобы форвард отрендеренного ботом списка не считался «новой» записью с нуля.
4. Я хочу, чтобы напоминание «купить хлеб завтра в 9» не зацепилось за старую заметку про хлеб и не отменило само себя.
5. Я хочу ответить на вопрос бота обычными словами («удали», «оставь оба»), а не искать кнопку.

## Проблема / зачем так

Закладку легко прислать дважды: ту же статью увидел в двух чатах, переслал боту его же
отрендеренный список, скопировал текст с эмодзи-чекбоксами. Без проверки в базе копилась
гора почти одинаковых записей, и поиск выдавал три варианта одного и того же.

Сравнивать «в лоб» по тексту мало. Пользователь шлёт перефразированный вариант — буквы
разные, смысл тот же. Или наоборот: форвардит сообщение бота, где к исходному тексту
приклеены служебные строки (`📋`, `Reply:`, `Выполнено:`) — буквальный текст похож, но
шум сбивает сравнение. Поэтому дедуп двухуровневый: сначала смотрит на смысловую близость
эмбеддингов, потом — на пересечение «очищенных» смысловых строк.

И есть обратная беда. Раньше сообщение «купить хлеб завтра в 9» уходило в ту же
дедупликацию и матчилось со старой заметкой «Записочка про хлеб». Бот показывал
`🔄 Похожая запись уже сохранялась: Записочка про хлеб`, напоминание при этом не
создавалось вовсе. Сейчас сообщения с reminder-намерением проходят мимо общего дедупа —
их точные повторы ловит отдельный механизм E15 (тот же текст + та же минута).

Дедупликация — целиком серверно-ботовая. Пользователь видит её только как сообщение от
бота в чате. На дубль бот пишет, например:

`🔄 Похожая запись уже сохранялась закладку: Статья про индексы от 18.05`
`Что делаем с новой? Ответь reply на это сообщение:`
`• открой — покажу старую`
`• удали — удалю новую (старая останется)`
`• обнови — заменю старую новой`
`• сохрани как новую — оставлю обе`

## Поведение (правила)

### Когда запускается проверка

Дедуп закладок срабатывает **после** AI-обработки сохранённой закладки, в воркере. Условия:
закладка в статусе `completed` или `partial`, есть чат для уведомления, и это **не**
task_list и **не** сообщение с reminder-намерением (см. ниже про пропуск).

Для незавершённых списков задач (task_list) общий дедуп откладывается: сначала бот
спрашивает «Сохранить как список?», и только если пользователь откажется — показывает
near-dup. Похожие незакрытые списки обрабатываются отдельным флоу слияния (БТ-4).

### Два уровня детекта дубля закладок

| Уровень | Что сравнивает | Порог | Когда работает |
|---------|----------------|-------|----------------|
| Pass 1 — эмбеддинг | косинусная близость векторов | cosine > **0.85** | только если у закладки есть embedding |
| Pass 1 — подтверждение текстом | при 0.85 ≤ cosine < 0.95 нужен ещё text overlap | overlap ≥ **60%** | для «средних» совпадений |
| Pass 1 — авто-дубль | очень высокая близость | cosine ≥ **0.95** | text-проверка не нужна, сразу дубль |
| Pass 2 — text overlap | пересечение смысловых строк без вектора | overlap ≥ **60%** | если raw_text ≥ 20 символов; ловит копипаст отрендеренного |

Pass 2 нужен, потому что копия отрендеренного ботом текста (эмодзи ☐/✅/⏰ → AI выдаёт
другой заголовок/summary → другой вектор) может вообще не совпасть по эмбеддингу, но
совпадает по строкам. Pass 2 берёт последние **20** закладок за **7 дней**.

### Что считается «смысловой строкой»

Перед сравнением строки нормализуются: убираются эмодзи и спецсимволы, bullet-маркеры
(`- • * — −`), нумерация (`1.`, `2)`), даты вида `DD.MM`, регистр приводится к нижнему.
Строки короче 3 символов и служебные строки бот-рендера (`📋`, `↩️`, `Reply:`, `Примеры:`,
`Выполнено:`, таймкоды `[HH:MM]`) выкидываются. Маркеры пунктов ☐/✅/⏰ при этом
**сохраняются** — это смысл, не шум.

Overlap считается симметрично: `max(совпало/|новое|, совпало/|старое|)`. Так дубль
распознаётся, даже если вокруг одного из текстов налеплено больше «нагрузки» (форвард
бот-рендера против голого списка).

### task_list ↔ task_list — сравнение по пунктам

Если обе записи — списки задач, overlap считается не по сырому тексту, а по множеству
текстов пунктов (`structured_data.tasks[].text`). Бот-генерённый заголовок и подсказки на
сравнение не влияют. Если хотя бы одна из двух — не список с пунктами, падаем обратно на
текстовый overlap по raw_text.

### Что НЕ участвует в дедупе

| Состояние закладки | Почему пропускается |
|--------------------|---------------------|
| `is_archived = true` | архив не загромождает сравнение |
| `ai_status` ≠ `completed`/`partial` | необработанные не сравниваем |
| reminder-намерение | действие во времени, не дубль заметки (E15 ловит точные повторы напоминаний) |

Reminder-намерение определяется по `reminder_decision.form` ∈ {`single_reminder`,
`composite_reminder`, `needs_button_choice`, `needs_hour`, `strong_intent_3button`}.
Такое сообщение не гоняется через general dedup. Точный повтор напоминания (тот же
нормализованный текст + та же минута `fire_at` среди pending) ловит `find_duplicate_reminder`
на всех путях создания — детали в [БТ-5](бт-05-напоминания.md).

### Текст алерта зависит от похожести

| Похожесть | Префикс в сообщении |
|-----------|---------------------|
| cosine ≥ 0.95 | `⚠️ Уже есть почти такая же <тип>: <заголовок> от <дата>` |
| 0.85 ≤ cosine < 0.95 (или text overlap) | `🔄 Похожая запись уже сохранялась <тип>: <заголовок> от <дата>` |

`<тип>` — «закладку» или «список». Дата — `от DD.MM` (если есть `created_at`), иначе без
даты. В silent-режиме перед отправкой алерта с исходного сообщения снимается 👀.

### Выбор пользователя (4 интента)

На алерт пользователь отвечает обычными словами — reply на сообщение или (в течение 5 минут)
просто следующим сообщением с ключевым словом. Интент распознаётся по словам **без LLM**:

| Интент | Ключевые слова | Что делает | Реакция |
|--------|----------------|------------|---------|
| open | открой, покажи, оригинал, старую | показывает старую запись, дубль НЕ трогает | превью старого + повторный вопрос что делать с дублем |
| delete | удали, убери, не нужен | удаляет новую, старая остаётся | `✅ Дубль удалён` (исчезает через 5с) |
| update | обнови, замени, перезаписать | переносит поля новой в старую, новую удаляет | `✅ Оригинал обновлён` (или ре-рендер списка) |
| save_new | сохрани, новую, копию, оставь | оставляет обе записи | `✅ Сохранено как новая закладка` (исчезает через 5с) |
| unknown | всё прочее | переспрашивает | «Не понял. Ответь: открой / удали / обнови / сохрани как новую» |

При неизвестном интенте сообщение пользователя удаляется, но Redis-состояние не чистится —
даётся ещё попытка. State живёт **24 часа**; «следующее сообщение без reply» работает
**5 минут** (ключ `pending_dedup`).

### Побочные эффекты «оставить обе» / «обновить»

«Сохрани как новую» и «обнови» дополнительно:
- возвращают 👍 на исходное сообщение, если в silent-режиме с него сняли 👀 (иначе фидбэка нет);
- если новая закладка — task_list, материализуют её (создают сообщение списка + пин + favorite): worker пропустил task_list-ветку из-за near-dup, теперь юзер подтвердил — список показывается;
- досоздают напоминания из persisted `reminder_decision` (near-dup заставил worker пропустить reminder-dispatch; иначе при «оставить обе» напоминания тихо терялись).

## Use Cases

### UC-3.1. Повторная ссылка/текст → выбор что делать

- **Предусловия:** у юзера уже есть похожая закладка за последние 7 дней.
- **Триггер:** прислал почти то же самое (та же ссылка, перефразировка, копипаст).
- **Основной сценарий:**
  1. Воркер обрабатывает новую закладку, находит дубль (эмбеддинг или text overlap).
  2. Бот шлёт алерт `🔄 Похожая запись уже сохранялась …` с 4 вариантами.
  3. Юзер reply `удали` → новая удаляется, `✅ Дубль удалён`.
- **Альтернативные сценарии:**
  - A1. `обнови` → поля новой переносятся в старую, новая удаляется, `✅ Оригинал обновлён`.
  - A2. `сохрани как новую` → остаются обе, `✅ Сохранено как новая закладка` + 👍 на исходное.
  - A3. `открой` → бот показывает превью старой и снова спрашивает что делать с дублем (старую не трогает).
  - A4. Ответ не похож ни на один интент → переспрос, ещё попытка.
- **Результат:** в базе либо одна запись, либо две — по выбору юзера.

### UC-3.2. Форвард отрендеренного ботом текста

- **Предусловия:** есть старая закладка/список.
- **Триггер:** юзер форвардит боту его же сообщение (с `📋`, `Reply:`, эмодзи-чекбоксами).
- **Основной сценарий:**
  1. Эмбеддинг может сильно отличаться (другой title/summary) → Pass 1 промахивается.
  2. Pass 2 нормализует строки (срезает служебный chrome) и ловит ≥60% пересечения.
  3. Бот показывает near-dup алерт.
- **Альтернативные сценарии:**
  - A1. Оба — списки задач → сравнение по пунктам, бот-заголовок/подсказки игнорируются.
- **Результат:** форвард не плодит «новую» запись без предупреждения.

### UC-3.3. Похожий незакрытый список задач

- **Предусловия:** есть незакрытый список с похожими пунктами за 7 дней.
- **Триггер:** юзер прислал новый список.
- **Основной сценарий:**
  1. Бот сначала спрашивает «Сохранить как список?» (offer).
  2. После подтверждения находит похожий незакрытый список (cosine > **0.7**).
  3. Шлёт `🔄 Похожий список <название> от DD.MM (N/M выполнено). Объединить новые задачи в него?` с кнопками `🔗 Объединить` / `📋 Отдельно`.
- **Альтернативные сценарии:**
  - A1. `🔗 Объединить` → пункты нового вливаются в старый, новое сообщение списка удаляется.
  - A2. `📋 Отдельно` → алерт убирается, оба списка остаются.
  - A3. Юзер отказался создавать список (offer «Нет») → показывается отложенный general near-dup.
- **Результат:** один объединённый список или два отдельных — по выбору. (Механика слияния — БТ-4.)

### UC-3.4. Напоминание не дедупится как закладка

- **Предусловия:** есть старая заметка про «хлеб».
- **Триггер:** «купить хлеб завтра в 9» (reminder-намерение).
- **Основной сценарий:**
  1. Воркер видит reminder-намерение → general dedup пропускается.
  2. Напоминание создаётся нормально, бот: `🔔 Напомню «купить хлеб» — 25.05 09:00`.
- **Альтернативные сценарии:**
  - A1. Точный повтор того же напоминания (тот же текст + минута) → E15: дубль не создаётся, `👌 Уже напомню …` (см. БТ-5).
- **Результат:** напоминание не теряется из-за совпадения с заметкой.

### UC-3.5. Ответ без reply (следующим сообщением)

- **Предусловия:** недавно (≤ 5 мин) показан near-dup алерт.
- **Триггер:** юзер пишет `удали` обычным сообщением, не делая reply.
- **Основной сценарий:**
  1. Бот находит `pending_dedup` для чата, распознаёт интент по ключевому слову.
  2. Применяет интент так же, как при reply (алерт правится через edit).
- **Альтернативные сценарии:**
  - A1. Прошло > 5 минут → `pending_dedup` истёк, сообщение обрабатывается как обычная закладка.
  - A2. Слово не распознано как интент → не перехватывается, идёт обычным флоу.
- **Результат:** не обязательно искать сообщение и делать reply.

## Корнер-кейсы (сводка)

| # | Кейс | Ожидаемое поведение |
|---|------|---------------------|
| 1 | cosine ≥ 0.95 | дубль сразу, без text-проверки, префикс `⚠️ Уже есть почти такая же` |
| 2 | 0.85 ≤ cosine < 0.95, но text overlap < 60% | НЕ дубль, пропускаем |
| 3 | Embedding отсутствует (partial после AI-fail) | Pass 1 пропущен, работает Pass 2 (text overlap) |
| 4 | raw_text короче 20 символов | Pass 2 не запускается |
| 5 | Форвард бот-рендера (эмодзи, `Reply:`, `📋`) | служебные строки срезаются, дубль ловится по смыслу |
| 6 | Оба — списки задач | сравнение по пунктам, заголовок/подсказки не влияют |
| 7 | Один список + одна обычная закладка | падаем на текстовый overlap по raw_text |
| 8 | Архивная похожая закладка | не участвует в сравнении |
| 9 | Похожесть только с разной «обвязкой» вокруг | симметричный overlap всё равно срабатывает |
| 10 | reminder-намерение похоже на старую заметку | general dedup пропущен, напоминание создаётся |
| 11 | Точный повтор напоминания (текст + минута) | E15: дубль не создаётся, `👌 Уже напомню` (БТ-5) |
| 12 | Незакрытый похожий список (cosine > 0.7) | предложение `🔗 Объединить` после offer |
| 13 | Закрытый похожий список | merge-alert не показывается (нужен хотя бы 1 невыполненный пункт) |
| 14 | Двойной тап `🔗 Объединить` | GETDEL состояния защищает: второй тап → «Предложение устарело» |
| 15 | Неизвестный ответ на алерт | переспрос, состояние не чистится, ещё попытка |
| 16 | Ответ без reply через > 5 мин | `pending_dedup` истёк, обычная закладка |
| 17 | «открой» | показ превью старого, дубль НЕ удаляется, повторный вопрос |
| 18 | «сохрани как новую» для task_list | новый список материализуется (сообщение+пин), напоминания досоздаются |
| 19 | Merge упал | `Не удалось объединить. Оставлю оба списка.`, обе записи живы |
| 20 | Update упал | `Не удалось обновить. Оба сохранены.` |
