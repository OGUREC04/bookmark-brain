# БТ-6: Напоминания × Списки задач (Phase 2.6/2.7)

> Бизнес-требования · living-doc «как ведёт себя сейчас»
> Обновлено: 2026-05-24

## Назначение и границы

Покрывает связку «список задач + напоминания»: как сообщение с задачами и датами
маршрутизируется в одну из форм, 3-кнопочный выбор, авто-создание напоминаний по
пунктам, каскад при редактировании списка (бот и Mini App).

НЕ покрывает: одиночное напоминание и разбор времени — [БТ-5](бт-05-напоминания.md).
Механику самих списков (чекбоксы, пин, /lists) — [БТ-4](бт-04-списки-задач.md).

## Пользовательские истории

Как пользователь:
1. Я хочу написать «завтра контрольная в 9, в пятницу зачёт в 18» и получить напоминание на каждую задачу, не создавая их по одному.
2. Я хочу, чтобы когда дата одна, а пунктов несколько, бот спросил — один общий будильник или список с напоминанием на пункт.
3. Я хочу дописать в список «врач 2026-05-20» и чтобы напоминание создалось само.
4. Я хочу перенести дедлайн пункта и чтобы связанное напоминание переехало за ним.
5. Я хочу редактировать дедлайны в Mini App и чтобы напоминания обновлялись так же, как из бота.

## Проблема / зачем так

Список задач и напоминание — разные сущности, но пользователь диктует их одним сообщением:
«завтра контрольная, в пятницу зачёт, ещё тесты». Раньше это либо дробилось неверно, либо
требовало ручного `/remind` на каждый пункт. И при редактировании списка напоминания
отвязывались — перенёс дедлайн в чеклисте, а будильник остался на старой дате.

Сейчас одно решение (`reminder_decision`) считает даты и пункты, выбирает форму, а правки
списка каскадом тянут за собой связанные напоминания — одинаково из бота и из Mini App.

## Поведение (правила)

### Формы и маршрутизация (date-count routing)

Router (`reminder_router.route()`) смотрит на число дат и пунктов:

| Вход | Форма | Что происходит |
|------|-------|----------------|
| **2+ дат** (на разных пунктах) | `task_list_with_reminders` | авто: по reminder'у на каждый dated-пункт. «✅ Создал N напоминаний» |
| **1 дата + 1 пункт** | `single_reminder` | авто: один reminder (см. БТ-5) |
| **1 дата + несколько пунктов** | `needs_button_choice` | 3 кнопки 📋 / 🔔 / ✕ — спрашиваем юзера |
| **дата без часа** | `needs_hour` | Reply-вопрос «во сколько?» |
| strong-intent без даты | `strong_intent_3button` | кнопки 🔔/📝/✕ (БТ-5) |
| нет дат / нет интента | `none` / `task_list_no_reminders` | напоминаний нет |

Решение сохраняется в `bookmark.structured_data.reminder_decision`; CAS-флаг
`reminder_decision_applied` защищает от двойного создания.

### 3 кнопки «📋 / 🔔 / ✕» (форма needs_button_choice)

Текст (с перечислением пунктов):
```
🤔 Одна дата, но несколько пунктов:
• <текст пункта 1>
• <текст пункта 2>
…

Как лучше?
• 📋 Список — оставлю чекбоксы, напомню только по пункту с датой
• 🔔 Напоминание — одно напоминание про весь текст
• ✕ — ничего, просто закладка
```
Каждый текст пункта HTML-экранируется; обрезается до 80 символов с «…»; показывается не более 8 пунктов (остаток: «…и ещё N»). Если пунктов нет — общий текст без списка.

| Кнопка | Действие | Итог |
|--------|----------|------|
| 📋 Список | `apply-decision form=task_list_with_reminders` | список с чекбоксами + reminder только на пункт с датой |
| 🔔 Напоминание | `apply-decision form=composite_reminder` (fire_at = первая дата) | одно напоминание на весь текст |
| ✕ | ничего | просто закладка, «Ок, без напоминания» |

Подтверждение: «✅ 📋 Список — создал N напоминаний» / «✅ 🔔 Напоминание — создал 1 напоминание».
State в Redis с anti-double-click (atomic pop); устаревший выбор → «Состояние устарело, попробуй через /todo».

### Reply «напомни …» на список

Reply на закреплённый task_list с «напомни …» / «сделай напоминание …» (БТ-5 триггеры) →
создаётся **composite reminder**, привязанный к этому списку. Время — из текста reply
(«напомни в пятницу в 18»), при отсутствии часа — спрашивает.

### Каскад при редактировании списка (`apply_cascade`)

Правка списка тянет связанные напоминания. Default-час для дедлайна без времени —
**09:00 в таймзоне юзера** (`/tz`).

| Правка пункта | Что с напоминанием |
|---------------|--------------------|
| Удалён пункт, где висел reminder | **cancel** |
| Добавлен пункт с дедлайном | **create** (fire_at = дедлайн + 09:00 user_tz) |
| Изменён дедлайн пункта | **reschedule** (переставляем fire_at) |
| Снят дедлайн у живого пункта | **cancel — только если** reminder создан каскадом (`source=cascade_added`); поставленный вручную не трогаем |

Подтверждение в боте — короткое, из `CascadeResult.summary()`: «✅ +1 напоминание» /
«✅ перенёс 1 напоминание» / «✅ отменил 1 напоминание». Если правка reminder'ов не
затронула — отдельного сообщения нет.

## Поведение — Mini App

Каскад **един** с ботом: при `PATCH /api/v1/bookmarks/{id}`, меняющем `structured_data`
списка, на бэке срабатывает тот же `apply_cascade`.

- Меняешь дедлайн пункта в редакторе списка (`TaskListEditor`) → PATCH → reminder
  создаётся / переставляется / отменяется по тем же правилам, что из бота.
- Снятие дедлайна у пункта → cancel связанного cascade-reminder'а.
- Активные напоминания видны в RemindersSheet (колокольчик в шапке), eager-load
  с title пункта (без N+1).

Mini App не показывает 3-кнопочный выбор — он живёт только в боте (форма `needs_button_choice`
рождается из NL-сообщения). Создание/правка из Mini App идёт через дедлайн пункта → каскад.

## Use Cases

### UC-6.1. Несколько дат → авто per-item

- **Триггер:** «завтра контрольная в 9, в пятницу зачёт в 18, ещё тесты Eltex»
- **Основной сценарий:**
  1. Router видит 2 даты → `task_list_with_reminders`.
  2. Список создаётся + по reminder'у на «контрольная» и «зачёт».
  3. «✅ Создал 2 напоминания». Пункт «тесты Eltex» без даты — просто в списке.
- **Альтернативные:**
  - A1. Две даты без разделителя/«что» («25 мая 1 июня экзамен») → **сейчас автоматически не разруливается** (целевой AI-fallback — bead `vou`, не зашиплен). Уходит в date-count routing по тому, что распознал парсер; при неоднозначности возможен лишний/неверный reminder. Известное ограничение.

### UC-6.2. Одна дата + много пунктов → 3 кнопки

- **Триггер:** «к пятнице: дочистить макеты, позвонить, оплатить»
- **Основной сценарий:**
  1. Router → `needs_button_choice`. Бот шлёт 3 кнопки.
  2. 📋 → список + reminder на пункт с датой.
  3. 🔔 → один composite reminder на весь текст (fire_at = пятница).
  4. ✕ → просто закладка.
- **Альтернативные:**
  - A1. Состояние выбора устарело (TTL/повторный клик) → «Состояние устарело, попробуй через /todo».

### UC-6.3. Одна дата + один пункт → авто single

- **Триггер:** «дочистить макеты в пятницу в 18»
- **Основной сценарий:** Router → `single_reminder` → один reminder сразу (детали — БТ-5).

### UC-6.4. Дата без часа → спросить время

- **Триггер:** «контрольная в пятницу» (час не указан)
- **Основной сценарий:**
  1. Router → `needs_hour`. Бот: «🕘 Дата есть, но не указано время. Во сколько напомнить?» + примеры.
  2. Reply «в 9» → reminder создаётся.

### UC-6.5. Reply «напомни» на список → composite

- **Предусловия:** в чате есть закреплённый task_list.
- **Триггер:** reply на него «напомни в пятницу в 18»
- **Основной сценарий:** создаётся composite reminder, привязанный к списку, на пятницу 18:00.
- **Альтернативные:**
  - A1. Без часа («напомни в пятницу») → спрашивает время.

### UC-6.6. NL-edit списка → каскад на напоминания

- **Предусловия:** список с напоминаниями на пунктах.
- **Триггеры и итоги (reply на список):**
  - «удали 2» (у пункта был reminder) → reminder **отменён** → «✅ отменил 1 напоминание».
  - «добавь врач 2026-05-20» → новый пункт + reminder на 20.05 09:00 → «✅ +1 напоминание».
  - «перенеси контрольную на пятницу» → fire_at связанного reminder'а **переставлен** → «✅ перенёс 1 напоминание».
- **Альтернативные:**
  - A1. Сняли дедлайн у пункта, reminder ставился **вручную** (не каскадом) → reminder НЕ трогаем.
  - A2. Правка не затронула ни одного reminder'а → отдельного сообщения про напоминания нет.

### UC-6.7. Mini App — правка дедлайна пункта → каскад

- **Триггер:** в редакторе списка меняешь/ставишь/снимаешь дедлайн пункта → PATCH.
- **Основной сценарий:** `apply_cascade` создаёт / переставляет / отменяет reminder по правилам выше — идентично боту.
- **Альтернативные:**
  - A1. Снял дедлайн у пункта с ручным reminder'ом → не трогаем (как UC-6.6 A1).

## Корнер-кейсы (сводка)

| # | Кейс | Ожидаемое поведение |
|---|------|---------------------|
| 1 | 2+ даты | per-item авто, без кнопок |
| 2 | 1 дата + N пунктов | 3 кнопки 📋/🔔/✕ |
| 3 | 1 дата + 1 пункт | single авто |
| 4 | Дата без часа | reply-вопрос «во сколько?» |
| 5 | 🔔 на 3-кнопке | composite на весь текст, fire_at = первая дата |
| 6 | Двойной клик по 3-кнопке | atomic pop → второй клик «состояние устарело» |
| 7 | Добавили пункт с дедлайном | create reminder на дедлайн + 09:00 user_tz |
| 8 | Перенесли дедлайн | reschedule fire_at |
| 9 | Сняли дедлайн (reminder каскадный) | cancel |
| 10 | Сняли дедлайн (reminder ручной) | НЕ трогаем |
| 11 | Правка не задела напоминания | без отдельного сообщения |
| 12 | Дедлайн в прошлом | reminder не создаётся (фильтр прошлого, см. БТ-5) |
| 13 | Правка дедлайна из Mini App | тот же каскад, что из бота (PATCH → apply_cascade) |
| 14 | apply-decision упал (3-кнопка) | «Не получилось создать напоминание», state уже снят |
| 15 | composite_fire_at в state повреждён | «Дата повреждена, попробуй создать через /remind» |
| 16 | Два одинаковых пункта с одной датой (per-item) | E15-дедуп на per-item НЕ применяется — возможны два reminder'а (known gap) |
