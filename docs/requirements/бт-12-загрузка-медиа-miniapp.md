# БТ-12: Загрузка медиа из Mini App (голос и файлы)

> Бизнес-требования · living-doc «как ведёт себя сейчас»
> Обновлено: 2026-06-18 · Связано: ADR `docs/decisions/0011-miniapp-media-upload.md`, бриф `docs/prd/MINIAPP-MEDIA-UPLOAD.md`, код `backend/app/api/uploads.py`, `backend/app/worker/uploads.py`, `shared/media/`; фронт (отд. репо `bookmark-brain-miniapp`): `src/lib/recorder.ts`, `lib/api.ts` (`uploadMedia`), `screens/ComposeScreen.tsx` (полноэкранный захват «Новая мысль», заменил шторку QuickCreateSheet), `lib/compose.ts`
> Статус реализации: Backend (эндпоинт + воркер-джоба) — реализовано за тестами, не задеплоено. Mini App: 🎤 ГОЛОС (фаза 1, `ti0`) — РЕАЛИЗОВАНО за флагом `VOICE_UPLOAD` (в репо false / локально true), запись MediaRecorder → `POST /upload` → поллинг `ai_status`; 📎 документы (фаза 2) — пока `disabled`. ai_error: бэк его НЕ отдаёт в `BookmarkResponse` (модель пишет, схема нет) → фронт показывает ОБЩИЙ текст ошибки; точную причину — отдельным мини-тикетом бэку.

## Назначение и границы

Покрывает приём голосовых записей и файлов **из Mini App** (HTTP, не через Telegram): эндпоинт загрузки, фоновое распознавание/извлечение текста и передачу заметки в обычный конвейер. **Не покрывает:** приём тех же медиа в боте (это бт-01/бт-02 — там бот сам качает из Telegram), базовую AI-обработку и связи (бт-02, бт-11), поиск (бт-07). Распознавание речи и извлечение текста — общий код с ботом (`shared/media/`), поведение распознавания то же.

## Пользовательские истории

Как пользователь:
1. Я хочу надиктовать заметку прямо в приложении, не уходя в чат с ботом.
2. Я хочу приложить PDF/DOCX в приложении и получить из него заметку.
3. Я хочу, чтобы приложение не зависало на время распознавания — сохранил и листаю дальше.
4. Если распознать не вышло, я хочу понятный статус, а не вечный «загружаю».

## Проблема / зачем так

В боте голос и файлы работают: бот качает их из Telegram, распознаёт и шлёт готовый текст в backend. Mini App так не может — он живёт в браузере внутри Telegram и Telegram-сообщения не отправляет. Раньше кнопки 🎤/📎 в приложении были выключены с подсказкой «вложения — в боте». Теперь backend принимает медиа напрямую по HTTP, а распознавание (которое жило только в боте) вынесено в общий код, доступный и воркеру backend. Распознавание долгое, поэтому приложение не ждёт ответа: загрузка создаёт заметку-черновик и сразу возвращает её, а текст подъезжает фоном.

## Поведение (правила)

**Эндпоинт `POST /api/v1/bookmarks/upload`** (multipart/form-data, авторизация как у всех вызовов Mini App)

| Поле | Обяз. | Назначение |
|---|---|---|
| `file` | да | бинарь: аудио или документ |
| `kind` | нет | `audio` / `document`; если не передан — определяется по MIME/расширению |
| `caption` | нет | приписка, попадёт в начало текста заметки |
| `duration` | нет | секунды (для аудио — маршрут sync/async Yandex) |
| `title` | нет | заголовок (для документов — имя файла) |

| Событие | Ответ |
|---|---|
| Тип не распознан (ни аудио, ни документ) | `415`, заметка и объект в хранилище не создаются |
| Файл больше лимита (аудио 25 МБ / документ 20 МБ) | `413` (файл не читается в память целиком) |
| Пустой файл | `400` |
| Успех | `201` + `BookmarkResponse` сразу; заметка `ai_status` = `transcribing` (аудио) / `extracting` (документ), `source=miniapp` |

**Распознавание/извлечение (воркер `process_upload_task`)**

| Правило | Поведение |
|---|---|
| Источник файла | временно лежит в Object Storage (S3), воркер качает и в конце удаляет |
| Браузерный формат аудио (WebM/MP4) | транскодируется через ffmpeg в OGG Opus перед Yandex STT |
| Родной формат (OGG/MP3) | транскод пропускается |
| Документ | текст извлекается (PDF/DOCX/TXT/MD), как в боте |
| Приписка (`caption`) | подставляется в начало распознанного текста |
| После распознавания | `ai_status=pending` → заметка уходит в обычный конвейер (эмбеддинг/классификация/связи) — он не дублируется |
| Стоимость доп. логики связей | как в бт-11: 0 токенов сверх обычной обработки |

**Статусы для Mini App:** приложение опрашивает `GET /bookmarks/{id}` и видит `transcribing|extracting → pending → completed` (или `failed` + `ai_error` при сбое распознавания).

**Фазирование:** фаза 1 — голос (главный запрос). Документы — та же механика тем же эндпоинтом.

## Use Cases

### UC-12.1. Голосовая заметка из Mini App

- **Предусловия:** пользователь авторизован в Mini App; кнопка 🎤 включена (`ti0`).
- **Триггер:** пользователь записал голос и отправил.
- **Основной сценарий:**
  1. Приложение шлёт `POST /upload` с аудио.
  2. Backend проверяет тип и размер, кладёт файл в S3, создаёт черновик `ai_status=transcribing`, возвращает `201`.
  3. Воркер качает файл; если формат браузерный — транскодирует в OGG; распознаёт через Yandex.
  4. Пишет текст в заметку, ставит `pending`, отдаёт в обычный конвейер; объект из S3 удаляет.
  5. Приложение по опросу видит `completed` и показывает готовую заметку.
- **Альтернативные сценарии:**
  - A1. Длинное аудио (>30 сек) → Yandex async-путь через S3 (как в боте).
  - A2. Распознавание вернуло пусто/ошибку → `ai_status=failed`, текст пустой; приложение предлагает переотправить.
  - A3. Сбой S3/БД при обработке → воркер ретраит; на последней попытке ставит `failed` (заметка не зависает в `transcribing`).
- **Результат:** надиктованная заметка появляется без ухода из приложения.
- **Правила:** приложение не ждёт распознавания синхронно; статус — единственный канал прогресса.
  Заметки из Mini App (`source=miniapp`) **НЕ дублируются сообщением в чат бота** — пользователь уже
  в приложении (бот шлёт карточку только для контента, пойманного из чата). Заголовок голосовой
  заметки **генерит AI из распознанного текста**; имя файла (`voice.webm`) в заголовок НЕ попадает.

### UC-12.2. Документ из Mini App

- **Предусловия:** кнопка 📎 включена (`ti0`).
- **Триггер:** пользователь приложил PDF/DOCX/TXT/MD.
- **Основной сценарий:** как UC-12.1, но черновик `ai_status=extracting`, текст извлекается (не распознаётся), сохраняется число страниц.
- **Альтернативные сценарии:**
  - A1. Формат не поддержан / файл битый / PDF под паролем → `ai_status=failed` + `ai_error`.
  - A2. Документ пустой (скан без OCR) → `failed`, понятная причина.
- **Результат:** заметка из текста документа.

### UC-12.3. Ошибка ещё на загрузке

- **Триггер:** неверный тип или слишком большой файл.
- **Основной сценарий:** backend отвечает `415` / `413` **до** создания заметки и записи в S3 — мусор не копится.
- **Результат:** приложение показывает тост, заметка не создаётся.

## Корнер-кейсы (сводка)

| # | Кейс | Ожидаемое поведение |
|---|------|---------------------|
| 1 | Браузер пишет WebM (Android) / MP4 (iOS) | транскод в OGG перед Yandex; пользователь разницы не видит |
| 2 | Голос уже в OGG/MP3 | транскод пропускается |
| 3 | Файл больше лимита | `413`, не читается в память целиком (защита от OOM) |
| 4 | Неподдерживаемый тип (картинка, zip) | `415` до создания заметки и до S3 |
| 5 | Распознавание упало (Yandex down, битый файл) | `ai_status=failed` + `ai_error`; заметка остаётся, можно переотправить |
| 6 | Транзиентный сбой (S3/БД) при обработке | ретрай; объект в S3 сохраняется между попытками; на последней попытке → `failed`, не зависает |
| 7 | Заметка удалена/уже обработана к моменту джобы | джоба пропускает (идемпотентность); осиротевший объект в S3 чистится |
| 8 | Приписка к голосу/файлу | подставляется в начало текста заметки |
| 9 | Очень длинное аудио | поднят таймаут джобы (5 мин); экстремально длинные — риск таймаута |
| 10 | Создание черновика упало после записи в S3 | объект из S3 удаляется (не остаётся сирота) |
| 11 | Параллельные/повторные загрузки | `source_message_id` пуст → дедуп-индекс не срабатывает, дублей по конструкции нет |
| 12 | Документ без расширения, но с валидным MIME | тип берётся и из MIME, не только из имени → распознаётся, не падает в `failed` |
| 13 | Очень длинная приписка / имя файла | обрезаются (приписка 50k, заголовок 500) — защита от раздувания LLM-обработки и БД |
| 14 | Сбой постановки в очередь после распознавания | заметка не коммитится в `pending` до успешной постановки → ретрай переобработает, не зависает |
