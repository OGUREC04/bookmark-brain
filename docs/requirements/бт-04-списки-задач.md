# БТ-4: Списки задач (task lists)

> Бизнес-требования · living-doc «как ведёт себя сейчас»
> Обновлено: 2026-05-24
> Связано: PRD [`TASK-LIST-EDITING`](../prd/TASK-LIST-EDITING.md) · код `backend/app/services/task_list_detector.py`, `task_list_editor.py`, `task_list_renderer.py`, `backend/app/worker/task_list_offer.py`, `bot/handlers/tasks/*`, Mini App (репо `bookmark-brain-miniapp`): `src/components/ds/TaskListEditor.tsx`

## Назначение и границы

Покрывает механику самих списков задач: как сообщение распознаётся как список,
оффер «Сделать список?», чекбоксы, дедлайны на пунктах, пин, NL-правка (reply),
команды (`/todo`, `/lists`, `/clearlists`, `/unpin`) и редактирование списка в
Mini App.

НЕ покрывает: каскад напоминаний при правке списка, авто-создание напоминаний по
датам пунктов, 3-кнопочный выбор «📋/🔔/✕», reply «напомни …» на список — это
[БТ-6](бт-06-напоминания-списки.md). Дедупликацию похожих списков — [БТ-3](бт-03-дедупликация.md)
(здесь только то, что оффер прокидывает похожий список в post-confirm alert).

## Пользовательские истории

Как пользователь:
1. Я хочу написать «купить хлеб, молоко, яйца» и получить аккуратный список с галочками, не подбирая команду.
2. Я хочу, чтобы бот спросил «это список?» — а не молча превращал любое перечисление в чеклист.
3. Я хочу отмечать пункты одним тапом по кнопке и видеть прогресс.
4. Я хочу править список обычным языком в ответ на сообщение: «закрой 1, 3», «добавь хлеб», «удали 2».
5. Я хочу, чтобы список висел закреплённым сверху, а когда всё сделано — сам откреплялся.
6. Я хочу редактировать список прямо в Mini App: тапать чекбоксы, менять текст, добавлять и удалять пункты, ставить дедлайны.

## Проблема / зачем так

Перечисление дел и обычная заметка выглядят одинаково — «купить хлеб, молоко,
яйца» это и список покупок, и просто текст. Раньше бот либо плодил из любой
запятой чеклист, либо не распознавал список вовсе. Сейчас детект эвристический и
консервативный: маркеры в начале строк, ≥3 коротких пункта через запятую, явный
триггер («список:», `/todo`), плюс anti-false-positive — рекламные посты с
хэштегами и телефонами, длинные абзацы с нумерацией списком НЕ становятся.

И даже когда детект сработал, бот не создаёт список молча — шлёт оффер «Похоже на
список — сделать закреплённый?». Промах детекта не превращается в мусорный
закреплённый чеклист одним нажатием «Нет».

## Поведение (правила)

### Детект списка

Детектор (`detect()`) проверяет по порядку, не бросает исключений:

| # | Сигнал | Порог | Итог |
|---|--------|-------|------|
| 1 | Явный триггер в начале текста | startswith одного из триггеров | force=True, префикс снимается |
| 2 | AI сказал `item_type=action` + маркеры | ≥2 распарсенных пункта | список |
| 3 | Маркированные строки | ≥2 строк с маркером, средняя длина пункта ≤80 симв | список (даже если AI промахнулся) |
| 4 | Inline через запятую | ≥3 пункта, средняя длина ≤40 симв, без точек | список |

**Маркеры строк** (`BULLET_RE`): `-`, `•`, `*`, `—`, `−`, `1.`/`1)`, а также свои
рендер-маркеры `✅ ☐ ✔ ✗ ☑ ☒` (чтобы переслать боту свой же отрендеренный список).

**Явные триггеры** (force): «сделай список», «оформи список», «в список»,
«списком», «список:», «список задач», «задачи:», «задачи на», «todo:», «to do:»,
«to-do:», «чеклист:», «чек-лист:», «чек лист:», «план:», «план на», «купить:»,
«закупка:», «шоппинг:», «shopping:». При force список создаётся даже из одной
строки и без anti-false-positive фильтра — юзер сам попросил.

**Преамбула голосовой диктовки** снимается с начала: короткая (≤6 слов, ≤40 симв)
строка из слов «нужно/надо/сегодня/завтра/запиши/так/короче…» — это вводная, не
пункт. «Сделать отчёт» НЕ преамбула (есть контентное слово).

### Anti-false-positive (НЕ список)

Структура говорит «это пост/статья» → список не создаётся:

| Сигнал | Порог |
|--------|-------|
| Длинный пункт | любой пункт > 100 симв |
| Рекламные маркеры | ≥2 пунктов с хэштегом / ссылкой / телефоном / @mention / «подпишись, канал, чат» |
| Описательная прелюдия | > 25 слов до первого маркера |

### Оффер «Сделать список?»

Воркер не создаёт список сразу — шлёт оффер с двумя кнопками:

> 📋 Похоже на список задач — N пунктов. Сделать закреплённый список?
> _Его можно отмечать галочками и редактировать ответом на сообщение._

| Кнопка | callback | Действие |
|--------|----------|----------|
| ✅ Да | `tlc:<bid>` | создаёт + биндит + пинит список, ставит `is_favorite=true` |
| ✕ Нет | `tlx:<bid>` | `structured_data=None` — остаётся обычной закладкой |

State в Redis `task_list_pending:{chat}:{msg}`, TTL 1 час, читается атомарным
GETDEL (защита от double-tap). Если Redis недоступен / оффер не отправился —
fallback на прямое создание (список не теряется). Устаревший оффер → «Предложение
устарело», сообщение удаляется.

### Рендер списка

Формат сообщения (`render_task_list_text`):

```
📋 Список

☐ 1. купить хлеб
☐ 2. молоко · ⏰ 25.05
✅ 3. яйца

Выполнено: 1 из 3

💬 Ответь на это сообщение чтобы изменить список
```

- Чекбокс: `✅` (done) / `☐` (не done). Выполненный пункт — зачёркнут (`<s>`).
- Нумерация 1-based.
- Дедлайн пункта → ` · ⏰ ДД.ММ` (или `ДД.ММ ЧЧ:ММ` если есть час). Общий срок
  списка (`common_deadline`) — в шапке после «Список».
- Заметка пункта (`note`) — отдельной строкой `↳ …`.
- Строка «Выполнено: X из Y» — только если есть хоть один done.
- Подсказка снизу. В silent-режиме — компактная (2 строки с примерами reply),
  кнопок нет. В verbose — кнопки.
- Backend-рендер и bot-рендер обязаны давать одинаковый HTML (две точки правды
  синхронизированы).

### Кнопки списка (verbose)

| Кнопка | callback | Действие |
|--------|----------|----------|
| `☐/✅ <текст>` (по пункту, до 15) | `tg:<bid>:<idx>` | **toggle** пункта (инвертирует done) |
| ⏰ Срок | `tldm:<bid>` | меню общего срока списка |
| 🗑 Удалить | `td:<bid>` | удалить список целиком (bookmark + сообщение + unpin) |

Меню общего срока (`tlds:<bid>:<code>`): Всё сегодня / Всё завтра / За неделю /
Убрать сроки — ставит/снимает `common_deadline` на весь список.

Добавление/удаление/переименование пунктов через кнопки **намеренно нет** — это
делается reply-фразой (NL-правка).

### NL-правка списка (reply)

Reply на закреплённое сообщение списка свободной фразой. Сначала пробуется
fast-path (regex, без LLM), при промахе — LLM-редактор:

| Фраза | Действие |
|-------|----------|
| «закрой 1, 3» · «выполни 2 и 4» · «3 готово» · «сделал 5» · «гтв 7» · «done 10» · «✅ 1, 2» | done=true (идемпотентно, НЕ toggle), несколько индексов через `,`/`и` |
| «10 не готово» · «отмени 12» · «верни 3» · «сними 5» · «открой 2» | done=false |
| «всё готово» · «закрой все» | done=true для всех |
| «добавь хлеб» · «+ X» · «запиши X» · «внеси X» | новый пункт в конец |
| «удали 2» · «удали 1, 3» · «убери 2 и 4» · «- 5» | убрать пункт(ы) по номеру |
| «9 до завтра» · «3 пятница» · «9: до 08.05» | дедлайн на пункт |
| «удали список» · «удалить» · «убери список» · «снеси» | удалить список целиком |
| сложная/неоднозначная фраза | → LLM-редактор (apply_nl_edit) |

Fast-path даты: сегодня/завтра/послезавтра, «через N дней», «через неделю», дни
недели (ближайший будущий), `ДД.ММ[.ГГГГ]`. Невалидный индекс (вне диапазона) →
фраза не применяется, уходит в LLM.

**Защита от LLM-галлюцинаций:** «N не готово / отмени / удали» НЕ может увеличить
список; фраза без add-маркера НЕ может увеличить; рост > 5 пунктов отклоняется
(фикс бага «12 не готово → 13-й пункт»). При отклонении / непонятной фразе — `👎`
на reply + подсказка с примерами, артефакты неудачных попыток чистятся при
следующем успехе.

### Пин и позиция в чате

- При создании список пинится (`disable_notification`), сервисное «бот закрепил…»
  удаляется автоматически (`on_pin_service_message`, только для наших списков).
- После любой правки список **пересоздаётся внизу чата** свежим сообщением, если
  уехал вверх (`_rerender_at_bottom`); если уже последний — edit на месте.
- Когда **все** пункты выполнены — список НЕ перепинивается и текущий пин
  снимается (`_maybe_autounpin`). Само сообщение остаётся как итог.
- `/unpin` — открепить все закреплённые сообщения в чате (сами списки остаются,
  видны через `/lists`).

### Команды

| Команда | Действие |
|---------|----------|
| `/todo пункт1, пункт2` | принудительно создать список (force-триггер «сделай список:») |
| `/lists [N]` | список всех task_list'ов с прогрессом `done/total`, пагинация по 5 |
| `/clearlists` | архивировать ВСЕ списки (обратимо, с подтверждением) |
| `/unpin` | открепить все закреплённые сообщения |

## Поведение — Mini App

В DetailScreen список (`structured_data.type === "task_list"`) рендерится
**интерактивно** компонентом `TaskListEditor` (US-1..US-5 из PRD реализованы).

Каждая строка (`TaskRow`):
- **Чекбокс** слева (`role=checkbox`, зона тапа ~28×28) — тап тогглит `done`,
  текст зачёркивается, `selection`-хаптик.
- **Текст** — тап → инлайн-`<input>`, Enter/blur сохраняет, Esc отменяет, лимит
  280 симв. Пустой текст после правки → пункт удаляется (с undo).
- **Чип дедлайна** (если есть) ` 📅 ДД.ММ` — тап открывает календарь (`DatePickerSheet`),
  `×` снимает дедлайн.
- **На ховере/фокусе:** иконка `×` (очистить текст) + `⋮` (меню: копировать /
  напомнить-дедлайн / удалить).

Внизу — строка `+ добавить пункт`: тап → инпут, Enter добавляет и держит фокус
для следующего, Esc/blur пустого — отмена.

Удаление пункта → тост «пункт удалён» с «отменить» (4 сек, портал).

**Сохранение:** оптимистично, дебаунс 400 мс (серия тогглов = один PATCH),
весь массив `tasks` целиком через `PATCH /api/v1/bookmarks/{id}` со
`structured_data`. При провале — откат к last-known-good + error-хаптик + тост
«не сохранилось». На unmount pending-дебаунс флашится (не теряем последнее).

Бэкенд на PATCH со `structured_data` запускает reminder-каскад по дедлайнам
пунктов — это [БТ-6](бт-06-напоминания-списки.md), фронт просто шлёт массив.

## Use Cases

### UC-4.1. Детект списка и оффер

- **Триггер:** `купить хлеб, молоко, яйца` (или маркированные строки).
- **Основной сценарий:**
  1. Детектор: ≥3 коротких пункта через запятую → список.
  2. Воркер шлёт оффер `📋 Похоже на список задач — 3 пункта. Сделать закреплённый список?`
  3. **✅ Да** → список создаётся, пинится: `📋 Список / ☐ 1. купить хлеб / ☐ 2. молоко / ☐ 3. яйца`.
- **Альтернативные сценарии:**
  - A1. **✕ Нет** → остаётся обычной закладкой (`structured_data=None`).
  - A2. Оффер устарел (TTL / double-tap) → «Предложение устарело», сообщение удаляется.
  - A3. Redis недоступен → fallback: список создаётся напрямую без оффера.
  - A4. Воркер нашёл похожий список → после «Да» приходит dedup-alert «Объединить?» (БТ-3).
- **Результат:** один закреплённый task_list либо обычная закладка.

### UC-4.2. `/todo` — принудительный список

- **Триггер:** `/todo купить молоко, позвонить маме, к зубному`
- **Основной сценарий:** force-триггер → список создаётся (оффер тоже показывается).
- **Альтернативные сценарии:**
  - A1. `/todo` без аргументов → справка по формату.

### UC-4.3. Отметить пункт (кнопка / Mini App)

- **Триггер:** тап по кнопке `tg:<bid>:<idx>` (бот) или по чекбоксу (Mini App).
- **Основной сценарий:**
  1. `done` пункта инвертируется, список перерисовывается.
  2. Бот: список уезжает вниз свежим сообщением.
- **Альтернативные сценарии:**
  - A1. Все пункты стали выполнены → «Список выполнен ✅», пин снимается.
  - A2. Mini App: серия тогглов → один debounced PATCH; провал → откат + тост.

### UC-4.4. NL-правка reply'ем

- **Предусловия:** в чате закреплён список.
- **Триггеры и итоги (reply на список):**
  - «закрой 1, 3» → пункты 1 и 3 done.
  - «добавь хлеб» → новый пункт в конец.
  - «удали 2» → пункт 2 убран.
  - «3 до завтра» → дедлайн на пункт 3.
  - «всё готово» → все done → список откреплён.
- **Альтернативные сценарии:**
  - A1. Сложная фраза → LLM-редактор.
  - A2. Непонятная фраза / галлюцинация LLM → `👎` + подсказка с примерами, список не меняется.
  - A3. «удали список» → список удаляется целиком (unpin + delete bookmark + сообщение).

### UC-4.5. Дедлайн на пункте

- **Триггер:** reply «9 до пятницы» (бот) или чип/меню → календарь (Mini App).
- **Основной сценарий:** `deadline` пункта = ISO-дата, в рендере ` · ⏰ ДД.ММ`.
- **Альтернативные сценарии:**
  - A1. Mini App: снять дедлайн (`×` на чипе) → `deadline=null`.
  - A2. PATCH с дедлайном → reminder-каскад на бэке (БТ-6).

### UC-4.6. `/lists` — все списки

- **Триггер:** `/lists`
- **Основной сценарий:** нумерованный список task_list'ов с `done/total` и общим сроком, пагинация по 5, кнопки «📋 открыть» / «🗑».
- **Альтернативные сценарии:**
  - A1. Списков нет → «У тебя пока нет списков задач. Создай: /todo пункт1, пункт2».

### UC-4.7. Mini App — редактирование списка

- **Триггер:** открыл закладку-список в DetailScreen.
- **Основной сценарий:** тоггл / правка текста / добавление / удаление / дедлайн — всё оптимистично, debounced PATCH массива целиком.
- **Альтернативные сценарии:**
  - A1. Пустой текст после правки → пункт удаляется + undo-тост 4 сек.
  - A2. PATCH провалился (500/network) → откат + тост «не сохранилось».
  - A3. Параллельная правка в боте → last-write-wins (PATCH шлёт весь массив).
  - A4. Закрытие Detail во время правки → флаш pending-дебаунса на unmount.

## Корнер-кейсы (сводка)

| # | Кейс | Ожидаемое поведение |
|---|------|---------------------|
| 1 | 2 пункта через запятую | НЕ список (нужно ≥3 inline) |
| 2 | Рекламный пост с 2+ хэштегами/телефонами | НЕ список (anti-false-positive) |
| 3 | Длинный нумерованный абзац (>100 симв в пункте) | НЕ список (статья) |
| 4 | Force-триггер из одной строки («список: купить хлеб») | список из 1 пункта (фильтр статьи отключён) |
| 5 | Переслан свой же отрендеренный список (`✅/☐`) | распознаётся обратно, шумовые строки (📋, Выполнено:) выкидываются |
| 6 | Голосовая преамбула «Сегодня нужно: …» | вводная снимается, пункты — после |
| 7 | «закрой 1» когда уже done | идемпотентно остаётся done (не toggle) |
| 8 | «12 не готово» (LLM хочет добавить пункт) | отклонено, список не растёт |
| 9 | «удали 99» (индекс вне диапазона) | не применяется (fast-path), уходит в LLM |
| 10 | Все пункты выполнены | пин снимается, «Список выполнен ✅» |
| 11 | Кнопки toggle только на первые 15 пунктов | да, длинные списки правятся reply'ем |
| 12 | Оффер: Redis упал | fallback — прямое создание, список не теряется |
| 13 | `/lists` пустой | подсказка про `/todo` |
| 14 | Mini App: удалил все пункты | список остаётся пустым `tasks: []` (авто-удаление пустой закладки на unmount из PRD — не зашиплено, known gap) |
| 15 | Mini App: текст пункта > 280 симв | обрезается до 280 при сохранении |
| 16 | Двойной тап «Да/Нет» на оффере | atomic GETDEL → второй клик «Предложение устарело» |
| 17 | Дублирующиеся пункты | разрешены, без проверок |
