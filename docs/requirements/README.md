# Бизнес-требования (БТ)

> Класс: living-doc · обновляется вместе с кодом (см. правило ниже)

БТ описывают **как система ведёт себя сейчас** — флоу, use cases, корнер-кейсы.
Это не «зачем строили» (это PRD) и не «как устроено внутри» (это SPEC/ARCHITECTURE).

## Жанры документации — не смешивать

| Документ | Отвечает на вопрос | Природа |
|---|---|---|
| **БТ** (`docs/requirements/`) | Как система ведёт себя сейчас | living-doc |
| PRD (`docs/prd/`) | Зачем строили, что решили | snapshot на момент решения |
| ADR (`docs/decisions/`) | Почему выбрали такой подход | вечное решение |
| `BOT-COMMANDS.md` | Что пользователь вводит | reference |
| `SPEC.md` / `ARCHITECTURE.md` | Как устроено внутри | техническое |

БТ **ссылаются** на PRD/ADR, а не копируют их.

## Правило обновления (living-doc)

Меняешь поведение фичи — правишь её БТ **в том же PR**. Иначе БТ начинает врать,
и весь смысл теряется. Это часть `docs-with-code`.

## Стиль

- Проза (Проблема/зачем) — конкретно, с реальными примерами бота, без ИИ-штампов
  (см. `D:\brain\wiki\концепции\анти-ai-писательство.md`).
- Структура (Use Cases, таблицы, правила) — телеграфно, без украшений.
- Скриншоты заменяем текстовыми примерами сообщений бота.
