# БТ-11: Связи между заметками (Connections)

> Бизнес-требования · living-doc «как ведёт себя сейчас»
> Обновлено: 2026-06-13 · Связано: PRD `docs/prd/CONNECTIONS-MVP.md`, эпик `docs/epics/connections-mvp.md`, код `backend/app/services/connections.py`
> Статус реализации: Backend + Bot — реализовано (за тестами, не задеплоено). Mini App (секция «Связано», таб «Граф») — контракты готовы, UI в задаче 9.

## Назначение и границы

Покрывает автоматическое связывание заметок по смыслу, просмотр связанных заметок в боте, семантический режим поиска и контракты графа. **Не покрывает:** базовую AI-обработку и эмбеддинги (бт-02), дедупликацию почти-дублей (бт-03), обычный поиск и просмотр (бт-07-поиск-просмотр) — связи лежат поверх них.

## Пользовательские истории

Как пользователь:
1. Я хочу, чтобы заметки связывались сами по смыслу, чтобы не размечать связи руками.
2. Открыв заметку, я хочу видеть похожие на неё, чтобы вспомнить контекст.
3. Я хочу искать по смыслу («инвестиции» находит «как поднять раунд»), а не только по точному слову.
4. Я хочу видеть граф своих заметок и ходить по нему, как в Obsidian.

## Проблема / зачем так

Заметки копятся плоским списком. Через пару недель пользователь не помнит, что сохранял, не находит нужное (поиск по словам не ловит «то же другими словами») и не видит, что две заметки про одно и то же. Связи строит сама система: при сохранении заметки бот ищет похожие по эмбеддингу и записывает рёбра. Раньше эмбеддинг считался только из ИИ-выжимки — на слабом GigaChat выжимка врёт, и связи врут вместе с ней; поэтому теперь основа эмбеддинга — реальный текст заметки, а ИИ-поля идут добавкой.

## Поведение (правила)

**Когда строятся связи**

| Событие | Что происходит |
|---|---|
| Сохранена/переобработана заметка | После персиста эмбеддинга воркер ищет top-k похожих и пишет рёбра (фоном, не блокирует подтверждение) |
| Связывание упало (ошибка БД и т.п.) | Best-effort: обработка заметки не падает, реакция 👍 ставится как обычно |
| Нет эмбеддинга (partial/`completed_no_embedding`) | Связи не строятся (нечего сравнивать) |
| Разовый бэкфилл (`backfill_bookmark_links`) | Строит связи для всех заметок с эмбеддингом; запускается вручную пользователем |

**Параметры связывания**

| Параметр | Значение | Примечание |
|---|---|---|
| Порог косинуса | **0.73** (откалибровано 2026-06-13) | ≠ порог дедупа 0.85; выше — мало связей, ниже — шум |
| k_store | 30 рёбер на заметку | баланс «Посмотреть все» vs объём БД |
| Направление | каноническое (одно ребро) | A↔B пишется один раз, читается с обеих сторон |
| Тип связи | `similar` (вес = косинус) | `manual`/`derived_from_space` — на будущее |
| Стоимость | 0 токенов LLM | чистый pgvector kNN |
| Скоуп | в пределах одного пользователя | кросс-юзерных связей нет |

**Источник эмбеддинга (общий с поиском и дедупом):** заголовок + реальный текст заметки (сырой ввод / транскрипт / добытая статья) + ИИ-поля (ключевые мысли, выжимка, теги), обрезка 8000 символов. Чанкинг длинных — вне MVP.

**Семантический режим поиска:** в `POST /api/v1/search/` поле `mode`. `hybrid` (по умолчанию) — смесь смысла и полнотекста с динамическими весами; `semantic` — строго по смыслу (вес 1.0/0.0). Если эмбеддинг запроса не получился — оба режима деградируют в полнотекст.

**Граф (два режима):** локальный (эго) — узел заметки и соседи на 1–2 шага, считается на лету; полный — строится по явному действию, тяжёлую раскладку считает клиент, координаты кэшируются на сервере, при росте числа заметок показывается флаг «устарел».

## Use Cases

### UC-11.1. Связь строится при сохранении

- **Предусловия:** у пользователя есть заметки с эмбеддингом; новая заметка обработана и получила эмбеддинг.
- **Триггер:** воркер завершил эмбеддинг новой заметки.
- **Основной сценарий:**
  1. Воркер ищет top-30 похожих с косинусом ≥ 0.75 в пределах пользователя.
  2. Для каждой пишет ребро `similar` (вес = косинус), направление каноническое.
  3. Повторная вставка того же ребра игнорируется (идемпотентно).
- **Альтернативные сценарии:**
  - A1. Похожих ≥ порога нет → рёбра не создаются, ошибки нет.
  - A2. Эмбеддинга нет → связывание пропускается.
  - A3. Ошибка записи → проглатывается, обработка заметки не страдает.
- **Результат:** в `bookmark_links` появились рёбра; «Похожие» доступны при просмотре.
- **Правила:** 0 токенов LLM; самосвязь исключена; архивные не связываются.

### UC-11.2. «🔗 Похожие (N)» в боте

- **Предусловия:** открыта карточка заметки (`view:{id}`), у заметки есть связи.
- **Триггер:** рендер карточки.
- **Основной сценарий:**
  1. Бот запрашивает связи; если N>0 — показывает кнопку «🔗 Похожие (N)».
  2. Пользователь жмёт кнопку → список топ-5 связанных, у каждой кнопка «открыть».
  3. Тап по связанной → её карточка (`view:{other_id}`); «◀️ Назад» возвращает к исходной.
- **Альтернативные сценарии:**
  - A1. Связей нет (N=0) → кнопка не показывается.
  - A2. Запрос связей упал → карточка показывается без кнопки (best-effort).
  - A3. Связи были, но исчезли к моменту тапа → «Похожих заметок пока нет».
- **Результат:** навигация по смысловым связям прямо в чате.
- **Правила:** N — истинное число связей (не размер показанного топ-5).

### UC-11.3. Семантический поиск

- **Предусловия:** у заметок есть эмбеддинги.
- **Триггер:** поиск с `mode=semantic`.
- **Основной сценарий:** запрос эмбеддится, ранжирование строго по косинусу; находит релевантное, где нет точных слов запроса.
- **Альтернативные сценарии:**
  - A1. Эмбеддинг запроса не получился → деградация в полнотекст.
  - A2. Ничего не найдено → пустой результат (не падает).
- **Результат:** результаты по смыслу, а не по словам.

### UC-11.4. Граф связей

- **Предусловия:** есть связи; (для полного графа) Mini App с табом «Граф».
- **Триггер:** открытие заметки (локальный) или кнопка «Построить граф» (полный).
- **Основной сценарий:**
  1. Локальный: сервер отдаёт узел заметки + соседей (≤ лимита), клиент рисует на лету.
  2. Полный: клиент считает раскладку, шлёт координаты на сохранение; следующее открытие — из кэша.
- **Альтернативные сценарии:**
  - A1. Граф ни разу не строился → пустой кэш, предлагается «Построить граф» (не ошибка).
  - A2. Добавились заметки с прошлой сборки → флаг «устарел», предложение пересобрать.
  - A3. Узлов больше лимита → выборка ограничивается (NFR-3).
- **Результат:** визуальная навигация по кластерам тем.
- **Правила:** тяжёлая раскладка — на клиенте (сервер лёгкий, тянет много юзеров).

## Корнер-кейсы (сводка)

| # | Кейс | Ожидаемое поведение |
|---|------|---------------------|
| 1 | У заметки нет похожих ≥ порога | Связей 0; кнопку «Похожие» не показываем |
| 2 | Первая заметка пользователя (корпус пуст) | Связей нет; не ошибка |
| 3 | Удаление заметки | Рёбра чистятся каскадом с обеих сторон |
| 4 | Заметка отредактирована | Связи MVP не пересчитывает (ребро по старому эмбеддингу) |
| 5 | Дубликат (косинус ~1.0) | Это связь `similar` с весом ≈1; дедуп — отдельный механизм (бт-03) |
| 6 | A и B сохранены одновременно | Одно каноническое ребро; гонка гасится UNIQUE + ON CONFLICT |
| 7 | Заметки на разных языках | Voyage multilingual — связь по смыслу кросс-язычно |
| 8 | Слабый ИИ исказил выжимку | Реальный текст в эмбеддинге держит смысл — связь не разваливается |
| 9 | Бэкфилл прогнан дважды | Рёбра не дублируются (идемпотентно) |
| 10 | Семантический поиск без эмбеддинга запроса | Деградация в полнотекст, не пустота-ошибка |
