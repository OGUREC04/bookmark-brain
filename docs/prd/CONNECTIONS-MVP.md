# PRD: Связи между заметками (Connections MVP)

## Executive Summary

Заметки копятся плоским списком и теряются. Эта фича строит **смысловые связи между заметками автоматически** — пользователь ничего не размечает руками (в отличие от Obsidian). При сохранении заметки бот находит похожие по смыслу (косинус по эмбеддингам) и связывает их. Пользователь видит «🔗 Связанное» при просмотре, ищет заметки **по смыслу** (а не по точному слову) и листает **граф связей** визуально в Mini App.

Архитектурно — **дешёвый гибрид**: вектор рождает связи → рёбра хранятся в нашем же PostgreSQL (`bookmark_links`) → двусторонность, обход и визуализация делаются обычным SQL + фронтендом. **Без Neo4j и GraphRAG** (обоснование — раздел Architecture, ресёрч 2026-06-01). Опирается на уже существующие pgvector + эмбеддинги Voyage voyage-3 + HNSW + прототип поиска похожих в дедупликаторе → почти не требует новой инфраструктуры.

## Problem Statement

У пользователя 50–300+ заметок в плоском `/list`. Через пару недель:
- не помнит, что сохранил;
- не находит нужное (поиск по словам не ловит «то же, но другими словами»);
- не видит, что две заметки про одно и то же — теряет смысловые кластеры;
- перестаёт сохранять — «всё равно потеряется».

Теги и full-text поиск не решают: тег = keyword, не смысл; разрозненные теги (#идея/#идеи/#startup); поиск по слову не находит синонимичное. Нужна структура, которая **сама** видит «эти заметки связаны» и даёт по ним перемещаться.

## Solution

1. **Авто-связывание на сохранении.** Новая заметка → эмбеддинг (уже считается, теперь из **реального текста заметки + ИИ-полей**, см. AD-4) → kNN-поиск похожих по косинусу → запись рёбер в `bookmark_links`. 0 токенов LLM.
2. **Просмотр связей.** При открытии заметки — «🔗 Связанное (N)»: топ-5 + «Посмотреть все».
3. **Семантический поиск.** Поиск заметок по смыслу запроса (косинус), а не по точному слову. Поверх существующего гибридного score = α·cosine + β·fts.
4. **Граф связей (визуализация).** Два режима (см. AD-2):
   - **Локальный граф** (по умолчанию, как local graph в Obsidian): открыл заметку → её узел + соседи на 1–2 шага. Считается на лету на клиенте из рёбер, мало узлов — мгновенно, без кэша.
   - **Полный граф** — строится **по явному действию** «Построить граф»: раскладка считается один раз и **кэшируется** (координаты узлов сохраняются), дальше — рендер готового. Баннер «граф устарел — обновить?» при накоплении новых заметок. Не пересчитывается на каждый вход.
5. **Разовый бэкфилл.** Существующие заметки переэмбеддятся новым текстом (реальный текст + ИИ-поля) и связываются один раз при релизе.

## User Stories

**US-1 — Связи строятся сами**
Как пользователь, я хочу, чтобы заметки связывались автоматически по смыслу, чтобы не размечать связи руками.
*Acceptance:* при сохранении заметки, у которой есть похожие (косинус ≥ порога), создаются рёбра; в просмотре заметки видно «🔗 Связанное».

**US-2 — Вижу связанное при просмотре**
Как пользователь, открыв заметку, я хочу видеть связанные, чтобы вспомнить контекст.
*Acceptance:* в боте под заметкой кнопка «🔗 Похожие (N)» (только если N>0); в Mini App — секция «Связано» с топ-5 + «Посмотреть все».

**US-3 — Ищу по смыслу**
Как пользователь, я хочу искать заметки по смыслу, чтобы находить «то же, но другими словами».
*Acceptance:* запрос «инвестиции в стартапы» находит заметку «как поднять раунд», где этих слов нет; результаты ранжированы по релевантности.

**US-4 — Листаю граф связей**
Как пользователь, я хочу видеть граф своих заметок и перемещаться по нему, чтобы исследовать кластеры тем.
*Acceptance:* в Mini App есть таб «Граф»: узлы = заметки, линии = связи; тап по узлу открывает заметку и подсвечивает её связи; граф не виснет на сотнях узлов.

**US-5 — Все связи под рукой**
Как пользователь, я хочу при необходимости раскрыть полный список связей, а не только топ-5.
*Acceptance:* кнопка «Посмотреть все связи» разворачивает полный список (отсортирован по весу).

## Functional Requirements

- **FR-1.** Текст эмбеддинга = **реальный текст заметки (заголовок + сырой текст / добытая статья) + ИИ-поля (ключевые мысли, теги)**, обрезка до 8000 символов (см. AD-4). На сохранении система ищет top-k похожих (k_store, дефолт 20–50) с косинусом ≥ `threshold` (старт 0.75) и пишет рёбра `kind='similar', weight=cosine`. Самосвязь исключена.
- **FR-2.** Связи **двусторонние**: при просмотре любой из двух заметок связь видна (запрос `from_id = X OR to_id = X`).
- **FR-3.** Показ ограничен топ-5 по весу; «Посмотреть все связи» отдаёт полный список.
- **FR-4.** Связываются **все типы** заметок (ссылка, статья, текст, пересланное) между собой.
- **FR-5.** Связи — **только в пределах одного пользователя** (`user_id`). Кросс-юзерных связей нет.
- **FR-6.** API: `GET /api/v1/bookmarks/{id}/related?limit=5` (+ `?all=true`).
- **FR-7.** Семантический поиск: добавить `mode=semantic` в существующий `POST /api/v1/search/` (semantic_weight=1.0; косинус по запросу).
- **FR-8.** API графа — два эндпоинта:
  - `GET /api/v1/graph/local?center={id}&depth=2` — эго-граф вокруг заметки, на лету из рёбер (лимит узлов).
  - `POST /api/v1/graph/build` — посчитать раскладку полного графа и **закэшировать координаты**; `GET /api/v1/graph` — отдать закэшированный граф + флаг устаревания (`stale`, `built_at`, `node_count`).
- **FR-9.** Mini App: таб «Граф». Локальный граф — мгновенно при открытии заметки. Полный — по кнопке «Построить граф»; рендер кэшированной раскладки (WebGL); баннер «граф устарел — обновить?» при росте числа заметок.
- **FR-10.** Bot: кнопка «🔗 Похожие (N)» под заметкой (только при N>0); тап → список связанных с tap-to-open.
- **FR-11.** Длинные заметки: реальный текст обрезается до 8000 символов перед эмбеддингом (как в текущем пайплайне). Чанкинг с усреднением — **вне MVP**.
- **FR-12.** Разовый бэкфилл: фоновая джоба **переэмбеддит** все заметки новым текстом (реальный текст + ИИ-поля) и строит связи. Идемпотентно (UNIQUE на ребро).

## Non-Functional Requirements

- **NFR-1 (токены).** Создание связей и семантический поиск = **0 токенов LLM** (чистый косинус/HNSW). Это жёсткое требование MVP.
- **NFR-2 (латентность).** Поиск связей на save не блокирует подтверждение сохранения — выполняется в worker (arq) фоном после эмбеддинга.
- **NFR-3 (граф не виснет).** Визуализация остаётся отзывчивой на ≥300 узлов (лимит/кластеризация на бэке при превышении).
- **NFR-4 (приватность).** Запросы связей/поиска/графа всегда скоупятся по `user_id`; утечки между пользователями невозможны.
- **NFR-5 (масштаб).** Корректно работает до ~100k заметок на пользователя на pgvector+HNSW (выше — вынос в спец-решение, вне MVP).

## Schema

Расширяем эскиз из SMART-SPACES-MVP (`bookmark_links` уже там описан):

```sql
CREATE TYPE link_kind AS ENUM ('similar', 'manual', 'derived_from_space');

CREATE TABLE bookmark_links (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    from_id UUID NOT NULL REFERENCES bookmarks(id) ON DELETE CASCADE,
    to_id UUID NOT NULL REFERENCES bookmarks(id) ON DELETE CASCADE,
    kind link_kind NOT NULL,
    weight FLOAT NOT NULL,                       -- cosine для similar
    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE (from_id, to_id, kind),
    CHECK (from_id != to_id)
);

-- Двусторонний запрос: индексы в обе стороны
CREATE INDEX idx_links_from ON bookmark_links(from_id, weight DESC);
CREATE INDEX idx_links_to   ON bookmark_links(to_id, weight DESC);
```

**Решение по двусторонности:** ребро пишем один раз (от новой заметки), а на чтении запрашиваем обе стороны (`from_id = X OR to_id = X`) и объединяем. Не дублируем рёбра — дешевле и нет рассинхрона. *(Альтернатива — симметричная запись с `CHECK(from_id < to_id)` — оставлена на случай, если запросы «обе стороны» окажутся горячими; решаем при реализации.)*

## Architecture Decisions

**AD-1. Вектор рождает связи, граф-фичи на SQL. Без Neo4j/GraphRAG.**
Обоснование (ресёрч 2026-06-01): «графы лучше» относится к multi-hop рассуждениям по сущностям, не к авто-связыванию похожего. Наша задача векторная. Рёбра в Postgres + SQL-обход в 2 шага закрывают двусторонность, навигацию и данные для визуализации. Граф-движок оправдан только при провале векторного поиска на multi-hop — для personal-notes порог не достигается. Прямой аналог (rhizome, Smart Connections) делает ровно это без графовой БД.

**AD-2. Визуализация графа — фронтенд + on-demand раскладка с кэшем.**
Рёбра берутся из `bookmark_links`. Два режима:
- **Локальный (эго) граф** — мало узлов, считается на лету на клиенте. Библиотека: **react-force-graph-2d** (canvas, лёгкая для мобильного Telegram WebView) или d3-force.
- **Полный граф** — тяжёлая часть это раскладка (позиционирование узлов). Считаем её **по явному действию** алгоритмом **Barnes-Hut / ForceAtlas2** (`graphology` + `graphology-layout-forceatlas2`) в фоне (web-worker или бэк), **сохраняем координаты узлов** и рендерим на **WebGL** (**Cosmograph** `@cosmograph/react` для 100k+ узлов, либо **Sigma.js + graphology**). Кэш инвалидируется по росту числа заметок (флаг `stale`).
Графовую БД (Neo4j) не вводим — рёбер в Postgres достаточно.

**AD-3. Параметры калибруются на бэкфилле.**
Стартовые: `threshold=0.75`, `k_store=20–50`, показ top-5. На бэкфилле строим распределение косинусов на реальных данных и фиксируем порог. Опционально mutual-kNN (ребро только если оба в top-k друг у друга) против шума. **Так как меняется общий эмбеддинг (AD-4), на бэкфилле же перепроверяем, что дедуп (порог 0.85) и гибридный поиск не деградировали; если поехали — правим пороги.**

**AD-4. Текст эмбеддинга: реальный текст + ИИ-поля (решение ревью 2026-06-13).**
Раньше эмбеддинг лепился только из ИИ-полей (title+takeaway+summary+key_ideas+tags) — связи становились заложником качества ИИ (на слабом GigaChat — глюки). Теперь **основа эмбеддинга — реальный текст заметки** (сырой текст / добытая статья), ИИ-поля (ключевые мысли, теги) — добавкой сверху. Так реальный контент держит смысл даже при плохой выжимке. Один общий эмбеддинг (дёшево на масштабе), обрезка до 8000 символов; чанкинг — вне MVP. **Блок-радиус:** эмбеддинг общий с поиском и дедупом → меняем `_build_embedding_text`, пороги перепроверяем на бэкфилле (AD-3).

## Token-cost analysis

| Операция | Стоимость | Примечание |
|---|---|---|
| Эмбеддинг новой заметки | уже в pipeline | не новая стоимость |
| Поиск похожих (kNN) на save | **0 токенов** | HNSW в Postgres |
| Семантический поиск | **0 токенов** | косинус |
| Граф / обход | **0 токенов** | SQL |
| Бэкфилл существующих | **0 токенов LLM** | переэмбеддинг всех заметок (one-time Voyage) + compute на kNN |
| *(отложено)* Типизация связи LLM | ~сотни токенов/пара | не в MVP |

## Edge Cases

| # | Кейс | Поведение |
|---|---|---|
| 1 | У заметки нет похожих ≥ порога | Связей 0; кнопку «🔗 Похожие» не показываем (не выдумываем) |
| 2 | Первая заметка пользователя (корпус пуст) | Связей нет; не ошибка |
| 3 | Очень длинная заметка | Реальный текст обрезается до 8000 символов перед эмбеддингом (чанкинг — вне MVP, FR-11) |
| 4 | Удаление заметки | `ON DELETE CASCADE` чистит рёбра с обеих сторон |
| 5 | Заметка отредактирована | MVP **не** пересчитывает связи (P2); рёбра остаются по старому эмбеддингу |
| 6 | «Популярная» тема (много похожих) | Хранение ограничено k_store, показ — top-5 + «все» |
| 7 | Дубликат (косинус ~1.0) | Это связь `similar` с weight≈1; дедуп — отдельный механизм, не конфликтует |
| 8 | Граф на сотнях узлов | Лимит узлов/кластеризация на бэке (NFR-3) |
| 9 | Заметки на разных языках | Voyage multilingual — связь по смыслу работает кросс-язычно |
| 10 | Бэкфилл на большом корпусе | Фоновая джоба батчами, идемпотентна (UNIQUE на ребро) |
| 11 | Гонка: две заметки сохраняются одновременно | UNIQUE(from_id,to_id,kind) + ON CONFLICT DO NOTHING |
| 12 | Семантический поиск без результатов | Пустой результат + подсказка; не падать |
| 13 | Граф ни разу не строился | Таб «Граф»: локальный работает сразу; для полного — кнопка «Построить граф» (пустой кэш → CTA, не ошибка) |
| 14 | Граф устарел (добавилось N заметок) | Баннер «граф устарел — обновить?»; пересчёт только по нажатию |
| 15 | Полный граф очень большой | Раскладка считается один раз и кэшируется; рендер на WebGL; при превышении лимита — кластеризация/лимит узлов |
| 16 | Слабый ИИ исказил выжимку | Реальный текст в эмбеддинге держит смысл (AD-4) — связь не разваливается на одной плохой выжимке |

## Success Criteria

- **SC-1.** ≥80% сохранённых заметок (у пользователя с >20 заметками) получают ≥1 осмысленную связь (проверка на ручной выборке после калибровки порога).
- **SC-2.** Семантический поиск находит релевантную заметку, которую full-text поиск по словам пропускает, в ≥70% тест-кейсов (golden set).
- **SC-3.** Создание связей и поиск не делают **ни одного** вызова LLM (NFR-1, проверяется логами/тестом).
- **SC-4.** Граф в Mini App открывается и остаётся отзывчивым на корпусе ≥300 заметок.
- **SC-5.** Бэкфилл проходит на проде без ручного вмешательства; рёбра не дублируются.

## Constraints & Assumptions

- Эмбеддинги Voyage voyage-3 (1024) уже считаются в pipeline; повторно не платим.
- Стек pgvector + HNSW (m=16, ef_construction=64) зафиксирован ADR 0004; не меняем.
- Worker arq доступен для фоновых джоб (связи на save + бэкфилл).
- Mini App — основной канал для графа (бот даёт только список «Похожие»).
- Порог 0.75 — стартовое предположение, **требует калибровки** на реальных данных (open question).

## Out of Scope

- **Типизация связей через LLM** («продолжает / противоречит / пример к») — следующая фаза; в MVP связь без «почему».
- **Пересчёт связей при редактировании** заметки — P2.
- **Ручное создание/удаление связей** пользователем (`kind='manual'`) — позже.
- **Graph-движок (Neo4j) / GraphRAG** — только если измерения покажут провал векторного поиска на multi-hop.
- **Чанкинг длинных заметок** (разбиение + усреднение эмбеддингов) — вне MVP; пока обрезка до 8000 символов.
- **Авто-пересборка полного графа** на каждый вход / по расписанию — вне MVP (строим по кнопке + подсказка об устаревании). Платная авто-пересборка — кандидат на монетизацию.
- **Шеринг групп заметок** (как общая база знаний) — отдельный будущий этап, бед `bookmark-brain-359`.
- **Связи между пользователями** — вне MVP (см. шеринг).

## Dependencies

**Есть (переиспользуем):**
- pgvector + HNSW-индекс на `bookmarks.embedding vector(1024)`.
- Pipeline эмбеддингов Voyage voyage-3.
- Прототип поиска похожих в дедупликаторе (`dedup_checker.find_near_duplicate`).
- Worker arq для фоновых джоб.
- Mini App (для секции «Связано» и таба «Граф»).

**Новое:**
- Таблица `bookmark_links` (+ миграция).
- Worker-джоба «build similar links on save» + джоба «backfill».
- API: `related`, `search?mode=semantic`, `graph`.
- Mini App: секция «Связано» + таб «Граф» (force-directed).
- Bot: кнопка «🔗 Похожие».

**Зависит от готовности:** Mini App MVP (для графа и секции связей).

- [ ] **Двусторонность.** Запрос «обе стороны» vs симметричная запись рёбер — решаем при реализации по горячести запроса.
