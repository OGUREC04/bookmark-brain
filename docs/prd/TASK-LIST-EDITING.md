# PRD · Task-list editing в Mini App

Статус: **draft**, ожидает утверждения.
Автор: Mini App-чат · 2026-05-21
Связано: `bookmark-brain-ntn` (UI cleanup), решение A2 из сессии 2026-05-20.

---

## 1. Проблема

В DetailScreen Mini App tasklist (`structured_data.type === "task_list"`) рендерится **read-only**: видно пункты и done-чек, но никак нельзя:
- отметить пункт выполненным;
- добавить новый пункт;
- удалить пункт;
- переименовать пункт.

Сейчас единственный способ редактировать список — писать боту reply («закрой 1, 3» / «добавь хлеб» / «удали 2»). Это работает, но юзер ожидает редактировать список в самой Mini App: открыл заметку → потогглил пункты → закрыл.

## 2. Решение

В DetailScreen рендерим список **интерактивно**: чекбокс тогглит done, инлайн-редактор позволяет менять текст пункта, кнопка «удалить» убирает пункт, поле «добавить пункт» внизу добавляет новый. Все изменения сохраняются через существующий `PATCH /api/v1/bookmarks/{id}` с `structured_data`.

Юзер-флоу:
1. Тапает на закладку-список → открывается DetailScreen.
2. Тапает чекбокс → пункт зачёркивается, прогресс пересчитывается, изменение летит на бэк (оптимистично).
3. Долгий тап на текст пункта → текст становится редактируемым → редактирует → blur/Enter сохраняет.
4. Свайп влево (или кнопка-корзина в конце строки) → удаляет пункт с подтверждением undo (snackbar 4 сек).
5. Кнопка `+ добавить пункт` в конце списка → инпут → Enter добавляет, Esc отменяет.

## 3. User Stories

### US-1 · тоггл чекбокса (A1, минимум)
**Как** пользователь со списком задач,
**я хочу** отметить пункт галочкой одним тапом,
**чтобы** не открывать чат с ботом и не писать «закрой 1».

Acceptance:
- [ ] Тап по чекбоксу или строке тогглит `done`.
- [ ] Визуально: галочка появляется, текст пункта зачёркивается, цвет → fg-3.
- [ ] Прогресс-бар в карточке и счётчик `done/total` обновляются мгновенно.
- [ ] Hapticfeedback `selection` на тап.
- [ ] Сохранение асинхронное (optimistic), при ошибке — revert + error-haptic + toast «не сохранилось».

### US-2 · редактирование текста пункта
**Как** пользователь,
**я хочу** поправить опечатку в пункте,
**чтобы** не пересоздавать список.

Acceptance:
- [ ] Долгий тап (≥350 мс) по тексту или тап по иконке «pencil» → текст становится contenteditable / `<input>`.
- [ ] Blur / Enter сохраняет, Esc отменяет.
- [ ] Пустой текст после редактирования — пункт удаляется (с undo).
- [ ] Максимум 280 символов на пункт (валидация).

### US-3 · удаление пункта
**Как** пользователь,
**я хочу** удалить ошибочный пункт,
**чтобы** не накапливался мусор.

Acceptance:
- [ ] Кнопка «×» в конце строки (visible on hover/active) ИЛИ свайп влево.
- [ ] После удаления — snackbar «удалён пункт N» с «отменить» 4 сек.
- [ ] Если последний пункт удалён — список не удаляется, остаётся пустым (см. open question).

### US-4 · добавление пункта
**Как** пользователь,
**я хочу** добавить пункт в существующий список,
**чтобы** не пересоздавать всё.

Acceptance:
- [ ] В конце списка строка-плейсхолдер `+ добавить пункт`.
- [ ] Тап → инпут, фокус сразу, Enter добавляет + остаётся фокус для следующего, Esc/blur пустого — отменяет.
- [ ] Новый пункт добавляется в конец, `done=false`, `deadline=null`.

### US-5 · дедлайн на пункте (resolved: in scope)
**Как** пользователь,
**я хочу** задать/изменить дедлайн у пункта прямо в Mini App,
**чтобы** не уходить в бот.

Acceptance:
- [ ] Если у пункта есть `deadline` — показываем чип с датой (read-only вид) рядом с текстом.
- [ ] Тап по чипу (или по «+ дата» если дедлайна нет) → пикер даты/времени → сохраняет в `deadline` (ISO).
- [ ] Очистка дедлайна возможна (кнопка «убрать» в пикере → `deadline=null`).
- [ ] Сохранение через тот же debounced PATCH всего массива `tasks`.
- [ ] Делается ПОСЛЕДНЕЙ итерацией (US-1..US-4 раньше).

### US-6 · переупорядочивание (out of scope)
Drag-and-drop / стрелки вверх-вниз — **не в этом PRD**, отдельная итерация.

## 4. Functional requirements

- Все операции работают офлайн-оптимистично: UI обновляется до ответа сервера, при ошибке откат.
- Изменения схлопываются в дебаунс 400 мс — серия быстрых тогглов = один PATCH.
- При параллельных правках с ботом (юзер в боте параллельно пишет «удали 2») — last-write-wins на уровне `tasks[]` целиком (`PATCH` шлёт весь массив).
- Чекбокс и текст — одна и та же UI-строка, разные зоны тапа (чекбокс ~28×28, остальное — редактирование).
- При полностью выполненном списке (`done === total && total > 0`) карточка в ленте получает done-tick (как сейчас).

## 5. Non-functional

- **Перформанс:** PATCH не чаще 1 раз/400 мс на список. Список до 50 пунктов рендерится плавно.
- **A11y:** чекбокс — `role=checkbox`, `aria-checked`. Удаление — `aria-label="удалить пункт N"`.
- **Хаптика:** `selection` на чек, `success` на add, `warning` на удаление.
- **Темизация:** только light (DS залочен).

## 6. API contract

**Существующий endpoint, без изменений в backend.**

`PATCH /api/v1/bookmarks/{id}` с body:
```json
{
  "structured_data": {
    "type": "task_list",
    "tasks": [
      { "text": "купить хлеб", "done": true,  "deadline": null },
      { "text": "пылесос",     "done": false, "deadline": "2026-05-22T18:00:00Z" }
    ]
  }
}
```

Возвращает обновлённый `Bookmark`. Шлём **весь массив `tasks`** при любой операции (toggle/add/delete/edit) — простота важнее минимизации трафика, списки короткие.

Если параллельный чат меняет схему `TaskListData` — синкаемся через `bookmark-brain-bne` (reminders 400) как пример того, как ловить рассинхрон.

## 7. Frontend архитектура

**Изменения:**
- `screens/DetailScreen.tsx` — заменить read-only `tasks.map` на компонент `<TaskListEditor>`.
- Новый компонент `components/ds/TaskListEditor.tsx`:
  - state: локальный `tasks: TaskItem[]`, синкается с props.bookmark.
  - debounced commit через `lib/api.ts::updateBookmark`.
  - Управление фокусом для add-input.
- `lib/api.ts::updateBookmark` — уже принимает `structured_data`, новых методов не нужно.

**Не трогаем:**
- ChatRow / BookmarkCard превью списка (`1/3 · хлеб, пылесос`) — текущий рендер из `bookmarkToChatRow` обновится автоматически после рефетча.

## 8. Edge cases

1. **Optimistic update upd до ответа:** юзер тогглит чек, PATCH ещё летит → второй тоггл другого пункта → дебаунс шлёт финальный state. ✅ ОК.
2. **PATCH провалился (500/network):** UI откатывает к last-known-good, тост «не сохранилось», error-haptic.
3. **AuthExpiredError:** PATCH вернул 401 после retry → выкидываем юзера на CTA «перезапусти Mini App» (есть в `lib/api.ts`).
4. **Параллельная правка в боте:** юзер в боте пишет «закрой 1», в Mini App в это же время тогглит 2. Last-write-wins: чей PATCH пришёл последним, тот победил. **Принимаем**, без оптимистичных конфликтов — списки короткие, частота правок низкая.
5. **Удаление последнего пункта списка:** список остаётся пустым (`tasks: []`), внизу строка «+ добавить пункт». **НО** (решение юзера 2026-05-23): если юзер удалил все пункты и **вышел из Detail ничего не добавив** → на unmount авто-удаляем закладку (`api.deleteBookmark`). Если успел добавить новый пункт — закладка сохраняется. То есть «пустой список при выходе = удалить закладку».
6. **Очень длинный текст пункта (>280):** валидация в onChange + ошибка-toast. Сохранение не отправляется.
7. **Пустой текст после редактирования:** автоматическое удаление пункта (как US-2).
8. **Дубликаты пунктов:** разрешены (юзер может специально написать одно дважды). Без проверок.
9. **Reorder в боте между сессиями Mini App:** при открытии DetailScreen берём свежий `bookmark` через рефетч, не доверяем кэшу.
10. **Эмодзи/RTL/специальные символы:** просто строка, без обработки. Telegram/iOS уже корректно рендерит.
11. **Конкуррентное удаление:** удалил пункт в Mini App → undo не успел нажать → таймаут PATCH'ит. Если юзер всё же нажал undo после PATCH — шлём новый PATCH с восстановленным пунктом. **Принимаем** двойной PATCH.
12. **Закрытие Detail во время правки:** при unmount флашим pending дебаунс синхронно. Не теряем последнее изменение.

## 9. Success criteria

**Definition of done:**
- US-1..US-4 acceptance закрыты.
- Tasks-editor работает на dev-боте (ngrok) на iOS + Android.
- При живом тесте: создаю список в боте → открываю Mini App → меняю 3 пункта (toggle/edit/add) → проверяю что в боте список тоже обновился после ребилда.
- 0 critical / 0 high от code-reviewer.

**Метрика (через neделю после shipping):**
- ≥50% юзеров со списками отредактировали хотя бы один список в Mini App, а не через бот.

## 10. Out of scope

- Drag-reorder пунктов (отдельный PRD при необходимости).
- ~~Дедлайны на пункт~~ → **в scope** (US-5, решение юзера 2026-05-23). Делается последней итерацией.

- `TaskListData` тип в `frontend/src/lib/api.ts` — `{ type: "task_list", tasks: TaskItem[] }`.
- `bookmarkToChatRow` уже корректно показывает прогресс — без изменений.
- DS-токены sage, fg-1..4, brand-tint, radius/chip — на месте.

**Что нужно:**
- Новый компонент `TaskListEditor.tsx`.
- Иконки: `pencil` (edit) и `×` (delete) — уже есть в `Icons.close`/`ExtraIcons.trash`, доп. SVG не нужны.

## 12. Open questions — RESOLVED (2026-05-23)

1. **Удаление: свайп vs кнопка?** → **Кнопка-крестик** на active-строке.
2. **Удалить весь список?** → **Хватит ⋯-меню** в шапке Detail.
3. **Пустой список?** → Пустой блок + «+ добавить пункт». **Дополнение:** если юзер вышел из Detail с пустым списком (все пункты удалены, ни одного не добавил) → закладка авто-удаляется на unmount.
4. **Дедлайн на пункте?** → **С редактированием** (не read-only). Вынесено в US-5, делается последней итерацией.
