> Статус: DRAFT → в работу · Создан: 2026-06-25
> Источник решений: understand-воркфлоу (4 линзы: backend-модель, фронт-экраны, ресёрч паттернов, корнер-кейсы) + 4 развилки, зафиксированы с владельцем 2026-06-25
> Связано: БТ-01 (захват), БТ-02 (AI-обработка), БТ-03 (дедуп), БТ-07 (поиск), БТ-11 (связи), тикет 0rn (inline-правка тела), тикет ti0 (голосовой захват)

## Проблема

Сейчас заметка — это **один статичный текст** (`bookmarks.raw_text`): записал и забыл. Но мысль живёт во времени: к «надо купить хлеб» через час хочется добавить «и молоко», вечером — «взял, но забыл масло». Сегодня для этого приходится либо переписывать тело заметки (теряя историю), либо плодить новые заметки (теряя связь). В итоге заметка не отражает, как мысль развивалась — а именно это делает личную базу живой, а не кладбищем.

**Примеры из жизни:**
- Заметка «идея для проекта» → через день дописал «поговорил с N, он скептичен» → через неделю «всё-таки делаю, вот первый шаг».
- «Купить подарок маме» → «смотрел свитер, дорого» → «нашёл на распродаже, беру».
- Рабочая мысль → серия уточнений по мере того, как разбираешься.

## Решение

Заметка становится **лог-перепиской со своими мыслями**: к ней можно **дописывать записи** со временем. Исходный текст заметки — её «шапка» (заголовок, саммари, теги, связи), под ней — лента дописок по времени, новая появляется внизу у поля ввода (как чат). Поле «дописать» (текст + голос) закреплено внизу экрана.

**Brain пока молчит** — он просто сохраняет, не отвечает. Но архитектура заложена так, чтобы будущие ответы Brain включились без миграции данных (поле `kind` у записи).

**Сознательно простой путь:** линейный лог (без веток/тредов), без ответов Brain, без напоминаний на отдельную запись. Но дописки **попадают в поиск и связи** (чтобы новые мысли не «терялись») и **можно дописывать голосом** — это явный выбор владельца, а не «дёшево по умолчанию».

## Ключевые решения (зафиксированы с владельцем)

| Развилка | Решение | Почему |
|---|---|---|
| **Хранение дописок** | Новая таблица **`note_entries`** (FK на заметку) | `raw_text` — несущая балка (эмбеддинг/дедуп/inline-правка/поиск), его не трогаем; JSONB-массив повторил бы боль `task_list` (гонки при синке между устройствами, нет правки одной записи, нет пагинации). Таблица = честный append одной строкой, FK-каскад, индекс по времени |
| **Исходный текст** | Остаётся в `raw_text` как «шапка»/запись #0, **без бэкфилла** | Миграция только структурная (CREATE TABLE), прод-прозрачная. Дублирование `raw_text` в `entries[0]` = два источника правды → рассинхрон при правке |
| **Поиск/связи дописок** | **Да, попадают** (отложенный/батч re-embed) | Цель Connections — «оживить базу»; невидимые в поиске дописки подрывают её. Brain при этом молчит — речь только про индекс |
| **Вид ленты** | **Как чат, новое внизу** + относительное время + дневные разделители | Метафора «переписка»; мышечная память Telegram (читаем сверху-вниз, пишем вниз) |
| **Правка/удаление дописок** | **Да**, на уровне записи; удаление — мягкое (`is_deleted`) | Согласуется с уже привычной inline-правкой тела (0rn); soft-delete не рвёт будущие ссылки Brain |
| **Голосовые дописки** | **В MVP** (запись → STT → дописка) | Явный выбор владельца. Добавляет работу: загрузка голоса в существующую заметку + статус распознавания на уровне записи |
| **Поле `kind`** | ENUM `user`/`brain`/`system`, в MVP всегда `user` | Бесплатный forward-compat (как `link_kind`/`scheduled_kind`): включение ответов Brain потом — без миграции схемы |
| **После создания заметки** | Открывать её «переписку» (`openDetail`) | В коде уже стоят 2 TODO(notes-as-conversations); сейчас уводит на список |

## User Stories

1. Как пользователь, открыв заметку, я хочу **дописать к ней новую мысль** (текстом или голосом), не переписывая старое и не плодя новую заметку.
   - *Acceptance:* поле ввода внизу экрана; отправка добавляет запись внизу ленты с меткой времени; заметка не дублируется.
2. Как пользователь, я хочу **видеть, как мысль развивалась** — ленту дописок по времени под шапкой заметки.
   - *Acceptance:* записи в хронологии (старое сверху, новое внизу), относительное время, разделители по дням при длинном логе.
3. Как пользователь, я хочу **поправить или удалить** опечатку/неактуальную дописку.
   - *Acceptance:* тап по записи = правка (как правка тела); есть удаление; удалённая исчезает из ленты.
4. Как пользователь, я хочу, чтобы **дописанные мысли находились поиском** и влияли на «Связи».
   - *Acceptance:* после дописки (с разумной задержкой) заметка находится по тексту дописки и пересобирает связи.
5. Как пользователь, создав заметку, я хочу **сразу попасть в её переписку**, чтобы продолжить мысль.
   - *Acceptance:* после сохранения открывается DetailScreen этой заметки (а не список).

## Функциональные требования

### FR-1. Лента (thread)
- DetailScreen показывает: **шапку** (заголовок, мета, теги, AI-саммари/key_ideas, тело `raw_text`, «Связано») + **ленту дописок** под ней.
- Порядок: хронология, старое сверху, новое внизу. Относительное время у записи (`formatRelativeDate`), дневные разделители (`DaySeparator`) когда лог > 1 дня.
- Удалённые (`is_deleted`) не показываются.

### FR-2. Дописка текстом
- Композер закреплён внизу DetailScreen (донор стилей — композер из `ComposeScreen`: textarea + авто-рост + `canSend`/`shouldExpandComposer`).
- `POST /bookmarks/{id}/entries {body}` → создаёт запись `kind='user'`, возвращает её; лента дорисовывает запись внизу, автоскролл к низу.
- Пустой текст (после `trim`) — отбой на клиенте (`canSend`) и на бэке (422).

### FR-3. Дописка голосом
- Та же кнопка 🎤, что в `ComposeScreen` (запись/таймер/волна/отмена), но цель — **дописка в существующую заметку**, а не новая.
- `POST /bookmarks/{id}/entries/upload` (multipart) → создаёт запись с `entry_ai_status='transcribing'`, возвращает её; воркер делает STT → заполняет `body`/`transcription`, статус `done` (или `failed`).
- Поллинг статуса — **на уровне записи** (не всей заметки): лента показывает «Brain слушает…» на этой записи, обновляет по готовности.

### FR-4. Правка / удаление записи
- `PATCH /bookmarks/{id}/entries/{entry_id} {body}` → правит `body`, ставит `edited_at`; в UI — тап по записи = inline-правка (как тело, авто-сейв на blur).
- `DELETE /bookmarks/{id}/entries/{entry_id}` → soft-delete (`is_deleted=true`).
- Правка/удаление записи **не гонит** материальность-гейт всей заметки (это отдельные строки, отдельные эндпоинты).

### FR-5. Индексация дописок (поиск + связи)
- Дописка/правка/удаление записи **планирует отложенный re-embed** заметки (debounce, не на каждый turn).
- Re-embed: `_build_embedding_text` расширяется — основа = `raw_text` + конкатенация **неудалённых** `note_entries.body` (хронологически), существующий cap 8000 символов. Также пересобирается `search_vector` (FTS) из той же конкатенации.
- После re-embed — пересчёт связей (`build_links_for_bookmark`, как сейчас).
- **classify/summary/title/теги НЕ пересчитываются** на дописку (дорого + UI-шум). Только эмбеддинг + search_vector + связи.

### FR-6. Навигация после создания
- `afterCreate` (текст) и `onCreated` (голос) в `App.tsx` → `openDetail(bm)` (открыть переписку заметки) вместо ухода на список. Поллинг `ai_status` и `replaceDetail` переиспользуются для свежести.

## Нефункциональные требования

- **NFR-1 (стоимость).** Дописка НЕ запускает LLM-классификацию. Re-embed — отложенный/батч (1 платный Voyage-вызов на «пачку» дописок, не на каждую). Цель — стоимость роста заметки ≪ стоимости новой заметки.
- **NFR-2 (IDOR).** Все операции с записями скоупятся через владение заметкой (404 если заметка не юзера), как `GET /related`.
- **NFR-3 (идемпотентность).** Пустые записи — 422. Двойной тап — блокировка кнопки на время запроса (`saving`-стейт, как в `ComposeScreen`). Опц. серверный дедуп «та же запись+заметка в окне ~5 сек» (защита от ретрая сети). Полный idempotency-key — вне MVP (Mini App не имеет `source_message_id`, бт-01 #93).
- **NFR-4 (синк между устройствами).** Append = INSERT строки (без гонки на массиве). Базовое ожидание категории (Telegram Saved Messages).
- **NFR-5 (клавиатура/скролл).** Закреплённый внизу композер + автоскролл к низу в Telegram WebView — известная зона граблей (iOS, safe-area, 100dvh). Проектировать аккуратно в дизайн-фазе.

## Модель данных

Новая таблица `note_entries` (миграция структурная, по образцу `add_bookmark_links`; FK CASCADE + индекс по родителю):

| Колонка | Тип | Null | Заметка |
|---|---|---|---|
| `id` | UUID PK | — | `gen_random_uuid()` |
| `bookmark_id` | UUID FK bookmarks ON DELETE CASCADE | NOT NULL | родительская заметка |
| `kind` | TEXT (ENUM-семантика, `create_type=False`) | NOT NULL | `user` (MVP) / `brain` / `system` — forward-compat |
| `body` | TEXT | NOT NULL | текст записи; ≤ 50 000 (как `raw_text`); для голоса = транскрипт |
| `created_at` | TIMESTAMPTZ | NOT NULL | `now()` |
| `edited_at` | TIMESTAMPTZ | NULL | проставляется при правке |
| `is_deleted` | BOOLEAN | NOT NULL | default `false` (soft-delete) |
| `media_file_id` | TEXT | NULL | голос: ссылка на исходное аудио |
| `transcription` | TEXT | NULL | голос: результат STT (дублирует `body` для аудита) |
| `duration` | SMALLINT | NULL | голос: длительность, сек |
| `entry_ai_status` | TEXT | NULL | голос: `transcribing`/`done`/`failed`; у текстовых = NULL |

Индекс: `ix_note_entries_thread` на `(bookmark_id, created_at) WHERE NOT is_deleted`.

**Инвариант:** оригинал = `raw_text` (всегда «запись #0», в `note_entries` НЕ дублируется); `note_entries` = только дописки (всегда после оригинала). Действует и для старых, и для новых заметок — нулевая миграция данных.

## API-контракты

| Метод | Путь | Тело | Ответ |
|---|---|---|---|
| GET | `/api/v1/bookmarks/{id}/thread` | — | `{ entries: [Entry], total }` — только дописки (шапку фронт берёт из существующего `GET /bookmarks/{id}`) |
| POST | `/api/v1/bookmarks/{id}/entries` | `{ body }` | `Entry` (kind=user) |
| POST | `/api/v1/bookmarks/{id}/entries/upload` | multipart (audio) | `Entry` (entry_ai_status=transcribing) |
| PATCH | `/api/v1/bookmarks/{id}/entries/{entry_id}` | `{ body }` | `Entry` (edited_at выставлен) |
| DELETE | `/api/v1/bookmarks/{id}/entries/{entry_id}` | — | 204 (soft-delete) |

`Entry = { id, kind, body, created_at, edited_at, voice?: { transcription, duration, ai_status } }`.

## Edge cases (корнер-кейсы)

| # | Кейс | Ожидаемое поведение |
|---|------|---------------------|
| 1 | Пустая дописка (только пробелы) | 422 на бэке + `canSend=false` на клиенте; запись не создаётся |
| 2 | Двойной тап «отправить» | Кнопка заблокирована на время запроса; опц. серверное окно дедупа ~5 сек |
| 3 | Дописка к старой заметке (без `note_entries`) | Работает: оригинал = шапка/запись #0, дописка — первая строка в `note_entries`; бэкфилла нет |
| 4 | Голосовая дописка, STT упал | `entry_ai_status='failed'`, в ленте «Не распознал»; запись остаётся (можно удалить/переписать текстом) |
| 5 | Голосовая дописка в обработке, юзер закрыл заметку | Запись с `transcribing` сохранена; при следующем открытии поллинг дотягивает результат |
| 6 | Правка записи | `edited_at` выставлен; в UI можно тихо пометить «изм.»; материальность-гейт заметки НЕ дёргается |
| 7 | Удаление записи | Soft-delete (`is_deleted=true`); исчезает из ленты и из конкатенации для эмбеддинга/поиска |
| 8 | Удаление заметки целиком | `note_entries` падают по FK CASCADE |
| 9 | Очень длинный лог (десятки записей) | Лента пагинируется/виртуализируется по `created_at`; эмбеддинг режется по cap 8000 (поздние записи могут не попасть в смысловой отпечаток — осознанная граница, windowing вне MVP) |
| 10 | Дописки и поиск | После debounce-reindex заметка находится по тексту дописки (FTS + семантика) |
| 11 | Дописки и дедуп (бт-03) | НЕ участвуют: дедуп живёт в воркере на тексте из чата, Mini App в нём не участвует by design — не регресс |
| 12 | AI-саммари (key_ideas) при росте лога | Стейл: шапка отражает только оригинал, дописки в classify не идут (MVP-граница). Зафиксировать; «пересобрать саммари» — кнопка/фаза потом |
| 13 | Связи бт-11 #14 при росте заметки | Новые связи добавятся, но вес уже построенных рёбер при переобработке НЕ пересчитывается (ON CONFLICT DO NOTHING) — приемлемо для MVP |
| 14 | Конкурентная дописка с двух устройств | Обе — отдельные INSERT, обе сохраняются (нет гонки на массиве) |
| 15 | Запись `kind='brain'`/`system` в MVP | Не создаётся (Brain молчит); если придёт — рендерится как обычный текст до включения двустороннего UI |

## Success criteria + гипотеза

- **Гипотеза:** возможность дописывать в заметку повышает возврат к базе. **Метрика:** доля заметок, получивших ≥1 дописку в течение 7 дней после создания; частота повторных открытий заметки с дописками vs без.
- **Корректность:** дописка находится поиском после re-index в ≤ окна debounce; правка/удаление консистентны (нет «призраков»); голосовая дописка доходит до `done`/`failed` без зависания.
- **Стоимость:** число LLM-classify на заметку не растёт с числом дописок (classify не вызывается на append).

## Out of Scope (MVP)

- **Ответы Brain** (генерация + двусторонний UI пузырей `kind='brain'`) — заложено архитектурно (`kind`), но не строится.
- **Напоминание на конкретную запись** — остаётся на заметку целиком (`bookmark_id`).
- **Ветвление/треды** (reply-to-entry, `parent_entry_id`) — линейный лог; YAGNI.
- **История версий записи** — только `edited_at`, без полного diff.
- **Пересборка AI-саммари/тегов по всему логу** — отдельная фаза (кнопка «переосмыслить»).
- **Ритуал ревью** (всплытие старых записей наверх по Karpathy) — продуктовая идея на потом.
