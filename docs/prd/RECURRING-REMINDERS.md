# Регулярные напоминания (Recurring Reminders) — PRD

> Статус: DRAFT → в работу · Создан: 2026-06-21 · Bead: `bookmark-brain-rnff`
> Связано: БТ-5 (Напоминания — ядро), БТ-6 (Напоминания × списки), ADR 0008 (scheduled_messages), code-explorer-карта подсистемы напоминаний

## Проблема

Текущие напоминания — одноразовые: `/remind полить цветы завтра в 10` сработает один раз. Но половина бытовых напоминаний по природе повторяющиеся: «полить цветы», «выпить таблетку», «отметиться на работе». Сейчас юзер вынужден заводить их заново каждый день — это ровно та трение, из-за которого напоминалками перестают пользоваться.

## Решение

Команда `/repeat <текст> каждый день в HH:MM` заводит **серию** — напоминание, которое повторяется ежедневно в заданное время (таймзона юзера). Доставка переиспользует существующий диспетчер: серия лишь «подкладывает» очередное одноразовое напоминание в общую очередь. При срабатывании — кнопки «✅ Ок» (принять это срабатывание, серия продолжается) и «🛑 Больше не напоминать» (остановить серию).

**Сознательно простой путь:** только явная команда, **без NL-автодетекта** регулярности из свободного текста. Только **ежедневно** (дни недели / месяцы — вне MVP, YAGNI).

## Ключевые решения (зафиксированы с владельцем)

| Развилка | Решение | Почему |
|---|---|---|
| Объём расписаний | **Только ежедневно** | покрывает «полить цветы»; парсер минимальный, мало корнер-кейсов |
| Ввод | **`/repeat` одной строкой** | как `/remind` (текст + расписание в одной команде) |
| Кнопки при срабатывании | **✅ Ок · 🛑 Больше не напоминать** | «продлить» бессмысленно — следующее уже завтра |
| Подсказка | **Всегда, одной строкой** под обычным напоминанием | нулевое состояние, максимальная заметность фичи |
| Авто-NL-детект регулярности | **Нет** | по явному запросу владельца — «не пытаться сами распознать» |
| Архитектура хранения | **Отдельная таблица `recurring_reminders`** | диспетчер `scheduled_messages` — fire-once (`pending→sending→sent`), не удаляет строки; мешать в него повторы = ломать концерны. Подробности — ADR (будет) |

## User Stories

1. Как пользователь, я хочу написать `/repeat полить цветы каждый день в 10:00` и получать это напоминание каждый день, не заводя его заново.
2. Как пользователь, при срабатывании я хочу одним тапом либо «принять» этот раз, либо остановить серию совсем.
3. Как пользователь, заводя обычное одноразовое напоминание, я хочу видеть подсказку, что бывают и регулярные — чтобы узнать про фичу.

## Спецификация команды

```
/repeat <текст> каждый день в <время>
```

- **Текст** — всё, что до фразы расписания. Пример: `полить цветы`.
- **Фраза расписания** (в конце): `каждый день` | `ежедневно` + `в <время>`.
- **Время:** `в 10`, `в 10:00`, `в 9:30`, опц. `утра`/`вечера`/`дня`/`ночи` (→ 24ч). Без времени — ошибка с примером.
- Парсинг + вычисление `next_fire_at` — **на бэкенде** (он знает `users.timezone`); бот шлёт сырой текст, бэк парсит и при неуспехе возвращает 422 с подсказкой-юзейджем.

**Подтверждение:** `🔁 Буду напоминать каждый день в 10:00 — «полить цветы»`.

## Модель данных

Новая таблица `recurring_reminders` (миграция, `down_revision = b1c2d3e4f5a6`):

| Колонка | Тип | Null | Заметка |
|---|---|---|---|
| `id` | UUID PK | — | `gen_random_uuid()` |
| `user_id` | UUID FK users ON DELETE CASCADE | NOT NULL | |
| `text` | TEXT | NOT NULL | ≤ 500 симв. (как `MAX_REMINDER_TEXT_LEN`) |
| `rule` | TEXT | NOT NULL | MVP всегда `daily`; строка ради forward-compat (`weekly:mon,tue` потом) |
| `hour` | SMALLINT | NOT NULL | 0–23, локальное время юзера |
| `minute` | SMALLINT | NOT NULL | 0–59 |
| `next_fire_at` | TIMESTAMPTZ | NOT NULL | UTC-инстант следующего срабатывания |
| `active` | BOOLEAN | NOT NULL | default `true` |
| `created_at` | TIMESTAMPTZ | NOT NULL | `now()` |
| `last_fired_at` | TIMESTAMPTZ | NULL | для наблюдаемости |
| `cancelled_at` | TIMESTAMPTZ | NULL | проставляется при 🛑 |

Индекс: partial `ix_recurring_next_fire` на `next_fire_at WHERE active`.

## Планировщик (materializer)

Новый cron `materialize_recurring` в воркере, **каждую минуту** (рядом с `scheduled_dispatcher`):

1. `SELECT … WHERE active AND next_fire_at <= NOW()`.
2. Для каждой строки — **CAS-advance**: `UPDATE recurring_reminders SET next_fire_at = <next>, last_fired_at = now() WHERE id=:id AND next_fire_at=:old`. Кто выиграл CAS (rowcount=1) — продолжает (защита от двойной материализации при наложении cron-ранов).
3. INSERT обычной строки `scheduled_messages`: `kind='reminder'`, `fire_at = old_next_fire_at`, `status='pending'`, `payload={"text":…, "source":"recurring", "recurring_id":…}`. Дальше её доставит штатный `scheduled_dispatcher`.
4. `<next>` = следующее ежедневное срабатывание строго позже «сейчас», вычислено из `hour:minute` в `users.timezone` → UTC (DST-safe: пересчёт каждый раз из локального времени).

**Доставка не дублируется:** диспетчер не трогаем, кроме выбора кнопок (см. ниже).

## Кнопки при срабатывании

Диспетчер при отправке смотрит `payload`: если есть `recurring_id` — строит **другие** кнопки:

```
✅ Ок            → rrok:<recurring_id>     (убрать это срабатывание; серия живёт)
🛑 Больше не      → rrstop:<recurring_id>   (active=false, отменить серию)
   напоминать
```

- `rrok` — редактирует сообщение («✅»), серверного действия не требует (`next_fire_at` уже сдвинут на материализации).
- `rrstop` — `DELETE /api/v1/recurring/{id}` (active=false, cancelled_at=now) **+** отмена ещё не сработавших `scheduled_messages` с этим `recurring_id` (status `pending`→`cancelled`). Редактирует сообщение → «🛑 Больше не напоминаю».

Заголовок самого срабатывания: `🔁 <текст>` (значок 🔁 отличает регулярное от разового 🔔).

## Подсказка под обычным напоминанием

В `_send_reminder_confirmation_with_chip` (bot/handlers/reminders/shared.py) и в worker-подтверждении одиночного напоминания (`reminder_decision._confirmation_text_single`) **всегда** добавляем хвост:

```
💡 Нужно повторять? <code>/repeat полить цветы каждый день в 9:00</code>
```

Списочные подтверждения (per-item / composite) — **не трогаем** (там контекст списка, подсказка была бы шумом).

## Корнер-кейсы

| # | Кейс | Поведение |
|---|---|---|
| 1 | Время уже прошло сегодня | первое срабатывание — завтра в это время |
| 2 | Нет времени в команде | 422 → «укажи время, напр. `/repeat полить цветы каждый день в 10:00`» |
| 3 | Нет фразы «каждый день/ежедневно» | 422 → юзейдж (это не регулярное) |
| 4 | Пустой текст (`/repeat каждый день в 10`) | 422 → «что напоминать?» |
| 5 | Дубль серии (тот же текст + время, active) | не плодим — возвращаем существующую: «Уже напоминаю каждый день в 10:00 — «…»» |
| 6 | Форматы времени: `в 10`, `в 10:00`, `в 9:30`, `в 6 вечера` | нормализуем в 24ч; `утра`=как есть, `вечера/дня`=+12 если <12, `ночи`=как есть |
| 7 | Невалидное время (`в 25:00`, `в 10:70`) | 422 → юзейдж |
| 8 | DST / смена tz юзера | `next_fire_at` пересчитывается каждый раз из локального `hour:minute` → сдвиг учитывается |
| 9 | Бот/воркер лежал дольше срока | срабатывает один раз с опозданием, дальше `next_fire_at` уходит в будущее — **пропущенные дни НЕ добиваются пачкой** |
| 10 | 🛑 при уже материализованном pending | серию гасим + отменяем ещё не сработавшие `scheduled_messages` (status pending→cancelled) |
| 11 | Юзер удалён | строки `recurring_reminders` удаляются по FK CASCADE |
| 12 | `/tz` изменён после создания серии | будущие срабатывания сдвигаются (пересчёт local→UTC при каждом fire) |
| 13 | Наложение cron-ранов | CAS на `next_fire_at` → ровно одна материализация |
| 14 | Очень длинный текст | обрезаем до 500 (как одноразовые) |
| 15 | `✅ Ок` | редактируем сообщение, серия продолжается (следующее завтра) |

## Out of Scope (MVP)

- Расписания кроме ежедневного (по дням недели, по числам, «каждые N дней»).
- Дата окончания серии («до 1 июля»).
- NL-автодетект регулярности из свободного текста.
- Редактирование серии (только создать + остановить).
- Снуз отдельного срабатывания.
- Управление сериями в Mini App (фаза позже).

## Acceptance

- [ ] `/repeat полить цветы каждый день в 10:00` → подтверждение `🔁 Буду напоминать каждый день в 10:00 — «полить цветы»`.
- [ ] В 10:00 (tz юзера) приходит `🔁 полить цветы` с кнопками `[✅ Ок][🛑 Больше не напоминать]`.
- [ ] На следующий день в 10:00 — приходит снова.
- [ ] `🛑` → «Больше не напоминаю», серия не срабатывает, pending-копии отменены.
- [ ] Под обычным `/remind`-подтверждением — подсказка-хвост всегда присутствует.
- [ ] Все корнер-кейсы 1–15 покрыты тестами; полный прогон зелёный; code-reviewer + security-reviewer чисто.
- [ ] БТ-5 дополнен секцией «Регулярные напоминания» (в рамках общего БТ-прохода после ручного теста).
