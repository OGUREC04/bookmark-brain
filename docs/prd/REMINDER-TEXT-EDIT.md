# Бриф для бэка: редактирование текста напоминания

**Статус:** бриф (frontend-driven), 2026-06-01
**Тикет:** bookmark-brain-8uu
**Связан:** PHASE-2.6-REMINDERS-X-TASKS, REMINDERS-MVP
**Кто инициатор:** Mini App (правка текста уже спроектирована в UI, но read-only из-за бэка)

> Этот документ описывает **что фронту нужно от бэка**, чтобы фича заработала.
> Решения по реализации (форма поля, валидация) — за бэком; ниже только контракт и контекст.

## Проблема

В Mini App на **переносе напоминания** (snooze) текст сделан **read-only**. Причина: текущий
`PATCH /api/v1/reminders/{id}` принимает только `fire_at` (`ReminderUpdate.fire_at`), а текст
напоминания лежит в `ScheduledMessage.payload["text"]` и эндпоинтом **не меняется**. Если бы
фронт слал новый текст — он бы **молча терялся**. Поэтому UI намеренно блокирует правку текста,
чтобы не врать пользователю.

Пользователь хочет **редактировать текст существующего напоминания** (а не только время).

## Что нужно фронту от бэка

1. **`PATCH /api/v1/reminders/{id}` должен принимать опциональный текст** и персистить его в
   `payload["text"]`.
   - Предлагаемый контракт (на усмотрение бэка — top-level поле либо merge в payload):
     ```
     ReminderUpdate {
       fire_at?: datetime   // как сейчас (snooze)
       text?: string        // НОВОЕ — новый текст напоминания
     }
     ```
   - Хотя бы одно из полей должно присутствовать. Можно слать **только `text`** (без смены времени)
     или **только `fire_at`** (как сейчас) или оба.
2. **Ответ** — `ReminderResponse` (уже содержит `payload`), чтобы фронт перерисовал новый текст
   без доп. запроса.
3. **Поведение при правке только текста:** `fire_at` не меняется, `status` остаётся `pending`.

## Контекст реализации (для бэка, не обязательность)

- ⚠️ **SQLAlchemy JSONB-мутация:** менять `payload` нужно через копию (`dict(reminder.payload)`)
  или `flag_modified`, иначе ORM не заметит in-place мутацию и **не запишет** (известный баг —
  см. вики `concepts/sqlalchemy-jsonb-mutation`). Симптом: лог «записал», в БД старый текст.
- Текст напоминания — **независимая копия** в `payload`, не ссылка на текст исходной закладки.
  Правка текста напоминания **не должна** менять связанную закладку (и наоборот). Фронт на это
  и рассчитывает.

## Edge-кейсы (важны фронту)

| # | Кейс | Ожидание фронта |
|---|------|-----------------|
| 1 | Правка только текста, время не трогаем | Текст сохранён, `fire_at` прежний, `status=pending` |
| 2 | Правка текста + времени одновременно | Оба применены |
| 3 | Пустой текст после trim | Бэк решает: reject (предпочтительно) — фронт покажет ошибку и не отправит |
| 4 | Напоминание уже сработало (`status != pending`) | Бэк: 409/422; фронт не показывает правку для не-pending |
| 5 | Очень длинный текст | Бэк решает лимит; фронт обрежет/покажет лимит, если бэк его озвучит |
| 6 | Чужое напоминание (не владелец) | 404 (как сейчас, user-scoped) |

## Открытые вопросы (на решение бэка)

- Форма поля: top-level `text` vs merge произвольного `payload`.
- Валидация пустого/длины текста.
- Дублировать ли правку текста в боте (сейчас snooze-флоу бота отдельный).

## Out of scope

- Правка текста **пунктов task_list** — у них свой редактор в Mini App (cascade через
  `PATCH /bookmarks` structured_data, тикеты `bbd`/`517`). Здесь речь о **самостоятельных**
  напоминаниях.

## Что разблокирует на фронте

После контракта Mini App снимет `readOnly` с textarea в `ReminderPickerSheet` на переносе и
включит `textChanged` в dirty-логику кнопки (сейчас исключён). Фронт-правка ~10 строк.
