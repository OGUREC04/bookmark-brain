# Бриф для бэка: редактирование текста заметок (любых)

**Статус:** бриф (frontend-driven), 2026-06-01
**Тикет:** bookmark-brain-0rn
**Связан:** TASK-LIST-EDITING (редактор пунктов), LINK-CONTEXT-EXTRACTION (z9q)
**Кто инициатор:** Mini App

> Описывает **что фронту нужно от бэка**. Глубина переобработки AI — **решение бэка** (см.
> «Открытые вопросы»). Фронту важен один сигнал — `ai_status` — и форма контракта.

## Проблема

Пользователь хочет **редактировать тело текста любой заметки**:
- дописать/исправить заметку к ссылке (например добавить свой комментарий к линку),
- отредактировать **расшифровку голосового** (STT ошибается),
- поправить обычную текстовую заметку.

Сейчас `PATCH /api/v1/bookmarks/{id}` (`BookmarkUpdate`) разрешает менять только
`title / is_favorite / is_archived / folder_id / structured_data`. **Тело текста
(`raw_text`, для голосовых — `transcription`) редактировать нельзя.**

## Что нужно фронту от бэка

1. **`PATCH /api/v1/bookmarks/{id}` должен принимать редактируемое тело текста.**
   - Минимум: добавить `raw_text?: string` в `BookmarkUpdate`.
   - Для голосовых уточнить: Mini App редактирует `raw_text` или `transcription`? Фронт ждёт
     **одно каноничное поле «текст пользователя»**, которое и показывает, и шлёт обратно.
     Если каноничен `raw_text` — ок; если для голосовых текст хранится в `transcription` —
     скажите, и фронт будет слать `transcription` для `content_type` голоса.
2. **Сигнал переобработки через `ai_status`.** Это ключевой контракт:
   - `BookmarkResponse.ai_status` фронт уже рендерит (состояние «Brain думает…»).
   - Если правка текста запускает фоновую переобработку (embedding/summary/теги) — бэк должен
     **выставить `ai_status` в обрабатывающее значение** в ответе на PATCH. Тогда фронт покажет
     «Brain переосмысливает…» и **обновит** закладку (poll `GET /bookmarks/{id}` или refetch),
     когда `ai_status` снова станет терминальным.
   - Если правка синхронна/без переобработки — `ai_status` остаётся терминальным, фронт сразу
     показывает финальное состояние.
   - **Фронту не важно КАК** бэк переобрабатывает — важно, чтобы `ai_status` честно отражал:
     «идёт обработка» vs «готово».
3. **Ответ** — обновлённый `BookmarkResponse`, чтобы фронт перерисовал текст (и `summary`/
   `key_ideas`/`title`, если они изменятся).

## Edge-кейсы (важны фронту)

| # | Кейс | Ожидание фронта |
|---|------|-----------------|
| 1 | Правка заметки к ссылке | `url` НЕ ломается; редактируется только текст-тело, не URL |
| 2 | Правка расшифровки голосового | Уточнить поле (`raw_text` vs `transcription`); сохраняется |
| 3 | Правка во время незавершённой обработки оригинала (`ai_status` ещё processing) | Бэк: разрешить/409. Фронт может заблокировать правку, пока идёт обработка, если бэк так решит |
| 4 | Пустой текст после trim | Бэк: reject (предпочтительно); фронт не отправит пустое |
| 5 | Очень длинный текст | Лимит на стороне бэка; фронт покажет, если озвучен |
| 6 | `task_list`-закладка | **Out of scope** — структурированный текст редактируется своим редактором (structured_data), не как свободное тело |
| 7 | Поиск после правки | После переобработки embedding — выдача по новому тексту; `search_vector` обновляется триггером БД автоматически |

## Контекст реализации (для бэка)

- `raw_text` тянет: `search_vector` (триггер БД, авто), `embedding` (воркер), AI-поля
  (`summary`/`key_ideas`/`title`/`takeaway`). При правке текста они **устаревают** относительно
  нового тела — отсюда вопрос глубины переобработки (ниже).
- Уже есть `POST /bookmarks/{id}/reprocess` и arq-pool в `bookmarks.py` — можно переиспользовать
  для постановки фоновой задачи.
- `update_bookmark` применяет поля через `setattr` по `exclude_unset` — добавление `raw_text`
  в схему технически достаточно для записи; вопрос только в триггере переобработки.

## Открытые вопросы (РЕШАЕТ БЭК — фронт подстроится через `ai_status`)

1. **Глубина переобработки после правки текста:**
   - только пере-embed (поиск точен; summary/теги/заголовок сохраняются как есть, без LLM),
   - полная переобработка (пере-классификация + embed + регенерация summary/тегов/заголовка),
   - без авто-переобработки (мгновенно; embedding стареет до ручного reprocess).
   Любой вариант ок для фронта — лишь бы `ai_status` честно отражал процесс.
2. **Sync vs async.** Если async — фронт ориентируется на `ai_status` + refetch. Если sync —
   PATCH возвращает финал.
3. **Каноничное поле текста** для голосовых (`raw_text` vs `transcription`).
4. **Конфликт** при правке во время незавершённой обработки (разрешать / 409).

## Что разблокирует на фронте

`DetailScreen` получит inline-редактирование тела текста (textarea как в task-редакторе:
auto-grow, brand-caret), кнопку «сохранить», состояние «Brain переосмысливает…» по `ai_status`.
Контракт типов в `src/lib/api.ts` (`Bookmark`) обновится в тот же заход.
