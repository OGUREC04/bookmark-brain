# Гайд по разработке BookmarkBrain

> Проверено: 2026-05-11 · класс: evergreen · конвенция обновления: `docs/README.md`

## Запустить локально

Двойной клик по `start.bat`. Скрипт сам поднимет базу и Redis в Docker, запустит сервер, фоновый воркер и dev-бота. Если что-то упадёт — смотри `startup.log` в корне проекта.

Остановить: `stop.bat`.

В первый раз на новой машине сначала `install.bat` — он создаст виртуальное окружение Python (это изолированная папка, куда ставятся библиотеки проекта, чтобы не мешались с системными), скачает все зависимости и применит миграции базы.

Два бота:
- `@bookmarkbrain_dev_bot` — для локальной разработки. Его токен лежит в локальном `.env`.
- `@N0teeBot` — боевой, на сервере. Его токен **никогда** не должен попадать в локальный `.env`.

Перепутать токены = твой dev-код побежит на живых юзерах. Проверяй перед каждым стартом.

## Сделать ветку и PR

Пять команд:

```bash
git checkout -b feat/коротко-что-делаю
# ...правишь код, прогоняешь pytest...
git add . && git commit -m "feat: что сделал"
git push -u origin feat/коротко-что-делаю
gh pr create --fill
```

Префикс коммита — `feat` (новая фича), `fix` (баг), `docs` (документация), `refactor` (переписал без новой логики), `chore` (мелочь типа обновления зависимостей). Это нужно чтобы потом по `git log` было видно тип изменения без чтения diff'а.

В `main` напрямую не коммитим. Имя ветки — латиницей, через дефис.

## Деплой

Одна команда с твоего ноута:

```bash
ssh root@5.181.109.142 "cd ~/bookmark-brain && git pull && ./deploy.sh"
```

Что она делает: заходит на сервер, забирает свежий код из main, пересобирает контейнеры. Контейнер — это упакованное приложение, которое работает одинаково и на твоём ноуте, и на сервере.

После деплоя проверь что бот живой:

```bash
ssh root@5.181.109.142 "docker compose -f ~/bookmark-brain/docker-compose.prod.yml logs bot --tail 20"
```

Если в логах нет ошибок и есть строчки про polling — всё ок.

## Куда класть тесты

Правило простое: **тест живёт там же где код, который он тестирует**.

| Код | Тесты |
|-----|-------|
| `bot/...` | `tests/test_*.py` (root) |
| `backend/...` | `backend/tests/test_*.py` |
| Что-то на живом Postgres | `backend/tests/integration/test_*.py` + `pytest.mark.integration` |

Run:
```bash
pytest tests/ backend/tests/                          # unit (default)
pytest backend/tests/integration/ -m integration      # требует docker-compose up
```

Не клади тесты в `tests/` для backend-кода — там conftest.py из root, не загружает backend env.

## Перед коммитом

1. **`pytest` в терминале — все зелёные.** Красные тесты в `main` ломают не только тебе.
2. **`.env` не в коммите.** Запусти `git status` — файла `.env` в списке быть не должно. Хук-блокировщик настроен, но мало ли.
3. **Поменял схему базы?** Сделал `alembic revision --autogenerate -m "что добавил"` и заглянул в сгенерённый файл — туда часто попадает мусор, который нужно убрать руками.
4. **Тронул классификатор, голос или эмбеддинги?** Прогнал ручной тест через dev-бота — отправил ему голосовое и текст, проверил что ответил. AI-пайплайн ломается тихо, тестами не всегда ловится.
5. **Поменял API или поведение бота?** Обновил `docs/SPEC.md` или `docs/ARCHITECTURE.md`, если изменение нетривиальное.

## Если в проде сломалось

Откат на предыдущий рабочий коммит:

```bash
ssh root@5.181.109.142
cd ~/bookmark-brain
git log --oneline -5            # покажет последние 5 коммитов
git reset --hard <SHA>          # SHA — это короткий хэш слева
./deploy.sh                     # пересобрать на старом коде
docker compose -f docker-compose.prod.yml logs bot --tail 50
```

Если упала только база (миграция кривая) — откатить одну миграцию назад:

```bash
docker compose -f docker-compose.prod.yml exec backend alembic downgrade -1
```

Если совсем плохо — полный перезапуск без потери данных:

```bash
docker compose -f docker-compose.prod.yml down
./deploy.sh
```

Данные базы лежат в Docker volume — `down` их не трогает.
