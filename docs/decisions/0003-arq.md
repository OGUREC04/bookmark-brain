# ADR 0003 — arq вместо Celery для фоновых задач

**Статус:** принято
**Дата:** 2026-03
**Связано с:** `backend/app/worker.py`, `backend/run_worker.py`

## Context

AI-обработка закладок (classify → embed → tag → dedup) занимает 2–10 секунд. Делать её синхронно в API-запросе бота — неприемлемо: бот должен отвечать реакцией 👀 за < 2 сек. Нужна очередь фоновых задач.

Кандидаты — Celery (классика Python), arq (async-native на Redis), RQ.

## Decision

Использовать **arq**.

## Consequences

**Плюсы:**
- **Async-native** — наш бэкенд весь на async SQLAlchemy + httpx, Celery бы заставлял писать sync-обёртки или жонглировать `asyncio.run_coroutine_threadsafe`.
- **Лёгкий** — один файл воркера, никаких отдельных брокеров кроме Redis (Redis у нас уже есть для state бота).
- **Простая модель** — функция = задача, без классов и сигнатур Celery.
- **Меньше зависимостей** — Celery тянет kombu, billiard и т.д.

**Минусы:**
- **arq CLI ломается на Python 3.14** — пришлось писать `run_worker.py` с `asyncio.run(worker.run())` вместо `arq backend.app.worker.WorkerSettings`. Зафиксировано в `D:\brain\claude-memory-compiler\knowledge\concepts\arq-python314.md`.
- **Меньше экосистема** — для Celery есть готовые dashboards (Flower), для arq — нет (есть `arq --check`, но и всё).
- **Меньше production stories** — Celery в индустрии 10+ лет, arq моложе.
- **Если понадобится cron на тысячи джобов в секунду** — придётся мигрировать. Но это не наш кейс в обозримые годы.
