# Bot UX — детали

> Проверено: 2026-05-23 · класс: evergreen · конвенция обновления: `docs/README.md`

## Команды

- `/start` — приветствие + создание пользователя
- `/todo пункты` — принудительно создать список задач
- `/list` — все закладки с inline-кнопками (📖 Открыть, 🗑 Удалить, пагинация)
- `/lists` — **только списки задач** (фильтр `structured_data.type=task_list`), с прогрессом `done/total` и общим сроком. Отдельная история от `/list`.
- `/unpin` — открепить **все** закреплённые в чате (Telegram `unpin_all_chat_messages`, не зависит от Redis-реестра — старые пины тоже снимаются). Сами списки остаются закладками.
- `/search запрос` — гибридный поиск (semantic + full-text + ILIKE fallback)
- `/random` — случайная закладка (ORDER BY RANDOM())
- `/stats` — статистика + топ теги
- `/clean` — удалить все сообщения бота в чате (не трогает закреплённые); `/clean all` — включая закреплённые
- `/reprocess` — батч-переобработка старых закладок
- `/help` — справка (авто-удаляется)

## Поведение

- Пересланные сообщения → сохраняются как закладки
- Обычный текст ≥15 символов → сохраняется автоматически
- Обычный текст <15 символов → запрос подтверждения (Да/Нет)
- Медиа без текста → "Добавь подпись"
- Медиа с подписью → сохраняется
- Live-прогресс обработки: 🔗→🤖→📊→🏷→✅ (воркер редактирует сообщение)
- Уведомление после обработки с inline-кнопками (Открыть, Удалить, Все закладки)

## Task list UX (Phase 2)

- **Редактирование списка — reply на сообщение со списком свободной фразой.**
  LLM (GigaChat через `POST /api/bookmarks/{id}/nl-edit`) применяет изменения к JSON: добавить/удалить/переименовать/проставить deadline/добавить note.
  Примеры: «добавь хлеб», «удали 2», «3 до пятницы», «к 1: нужен бездрожжевой».
- Клавиатура минимальна: чекбоксы-toggle + ряд `[⏰ Срок] [🗑 Удалить]`. Никаких «Дополнить» / «Не список» — всё через NL-reply.
- **Подтверждение перед созданием:** worker не создаёт/пинит список сразу — шлёт offer «Сделать список? ✅ Да / ✕ Нет» (зеркало reminder-offer). «Да» → bot создаёт+пинит+favorite; «Нет» → остаётся обычной закладкой (`structured_data=None`). Callbacks `tlc:`/`tlx:`, Redis `task_list_pending:{chat}:{offer_msg}` TTL 1ч.
- Шапка списка — нейтральная `📋 Список` (без AI-заголовка: галлюцинировал и шумел) + тег общего срока. Константа `LIST_HEADER`, синхронна в bot/backend рендерерах.
- Авто-pin + авто-favorite — по подтверждению. **Авто-unpin когда все пункты выполнены** (кнопкой и reply); slow-path rerender не перепинивает.
- Redis-map `task_list_msg:{chat_id}:{msg_id} → bookmark_id` (TTL 14 дней) разделяется между worker (пишет при создании) и bot (читает в reply-handler). См. `bot/state_store.py` и `_bind_task_list_message` в `backend/app/worker.py`.

### Слои UX при создании списка (НЕ сливать в один)

Порядок диктует mental-model юзера, не порядок в pipeline:

1. **Offer (подтверждение)** — всегда первым. «Сделать список? ✅ Да / ✕ Нет». Защищает от случайных пинов.
2. **«Да» → создаём+пиним, затем merge-alert** — если есть похожий незакрытый список (`similar`): «🔄 Похожий список — объединить?» (`dm:`/`dk:`). Это **post-confirm** шаг, отдельный от offer.
3. **«Нет» → обычная закладка, затем general near-dup alert** — если есть похожая закладка вообще (`general_dup`): «Похожая запись уже есть, что делать?» (reply: открой/удали/обнови/сохрани как новую).

Worker **пред-вычисляет** `similar` + `general_dup` и кладёт сериализованно в Redis `task_list_pending` — bot читает один JSON и действует, без доп. RPC. «Сохранить как новую» на general near-dup **материализует** список (создаёт+пинит), если bookmark был `task_list`.

### Голосовой список (STT preprocess)

Yandex Async STT (>30с) вставляет `[MM:SS]` на паузах — это **границы chunks, не пункты**. Пайплайн:

1. `strip_timestamps` — `[MM:SS]` → `\n` (сохраняем структуру chunks как строки).
2. `bot/services/voice_list.preprocess_voice_list` — группирует строки в пункты: цифра-в-начале → новый пункт, не-цифровая строка → континуация (склейка), голая цифра → склейка со следующей, короткая преамбула («Сегодня нужно») → дроп.
3. Backend `task_list_detector` **тоже** дропает ведущую преамбулу в forced-ветке (источник правды — на случай текста/форварда без bot-preprocess).

Голосовое сообщение-источник **не удаляется** (это контент, не дубль текста).

### Routing reply-на-бота

Reply на сообщение бота **никогда** не попадает в catch-all «Сохранить как заметку?». Catch-all `handle_text` имеет фильтр `~F.reply_to_message.from_user.is_bot` — reply-боту обрабатывают upstream reminder/tasks-роутеры явно. Reply на чужое/своё сообщение всё ещё может стать заметкой (без регрессии).

## Clean chat mechanics

- Все исходящие bot-сообщения трекаются через aiogram session middleware `TrackingMiddleware` в Redis `bot_msgs:{chat_id}` (ZSET, TTL 48ч — предел Telegram).
- Закреплённые сообщения помечаются в `bot_msgs_pinned:{chat_id}` и по умолчанию из /clean исключаются.
- Транзитные сообщения (подтверждения, ошибки, /help, подсказки «не нашёл список») авто-удаляются через 8 сек через `_ephemeral` хелпер.

## AI-обработка (воркер)

1. Fetch `<title>` из URL (httpx, fallback на summary)
2. GigaChat классификация → summary (рус), tags (рус), category, language
3. Voyage AI embedding (1024 dims)
4. Создание тегов (get-or-create)
5. Уведомление пользователя с временем обработки
